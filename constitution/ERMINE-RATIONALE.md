## RAT:R-STATE-09
LAW-3's promise is not "no two words touch a property" but "no *ambiguous* fight over a property"
— the thing that would force cascade-order tiebreaking. A base declaration plus an event-triggered
override is not ambiguous: the condition partitions when each applies (`ground` when idle,
`ground-hover` while hovered), so no winner has to be picked. So conditioned skin composes with
base skin without violating LAW-3 — the same reason the button's `:hover` background never fought
its base. Buttons never hit this because their state styling is CSS on a pseudo-class, not an
axis; the collision only appears when an interaction state (selection) is modelled as an axis. The
rule sanctions base-vs-conditioned; two conditioned words firing on the *same* event are left to
the cascade because their priority is a genuine design choice, not a universal the grammar can
rule. Resolves GAP-K6-skin-surface's B4 question. Surfaced by `how do buttons resolve this`.
Source: ADR-0006.

## RAT:R-STATE-10
R-STATE-07 kept interaction states bare because it saw them as condition-only predicates
(R-STATE-01). Skin carriers plus R-STATE-09 exposed a second use: a per-element skin override
that fires only under the condition (a tab raising its own ground on hover). The Monky evidence —
46 conditioned skin rules, hover dominant — shows those tones are per-element design choices, not
derivable from element kind, so the platform-first cascade cannot absorb them and leaving them
local hides real design in the stylesheet. The prefix promotes that intent into the class
paragraph; because the values are already carrier words, only the *condition* is new. It is a
prefix rather than a bare word because the condition scopes an existing declaration instead of
adding a value — the R-STATE-07 shape. Platform conditions need no backing (the browser owns
hover/active/focus), which is the seam with app-asserted selection (`selectable`, R-STATE-08); a
bare interaction predicate and a `hover:` scope can coexist without conflict. The paragraph
deliberately omits UA-supplied decoration such as the focus ring — redrawing what the platform
already provides is the anti-pattern — and verbosity stays bounded because the regular treatment
is one-to-two declarations. `over` and `float` were weighed as names and set aside: `over` loses
the hover/active/focus family coherence and is overloaded, `float` collides with the CSS property
and names elevation, not pointer presence. Surfaced by the conditioned-skin evidence pass.
Source: ADR-0007.

## RAT:R-STATE-11
The original `selection-treatment` axis offered two fixed levels (`selection-subtle`,
`selection-strong`) that wrote `--selection-bg/ink/outline` custom properties, hard-coded to one
hue and read by a sink. Wiring Monky's real selection revealed three mismatches at once: Monky
selects in its accent hue, not the axis's harmonic; it draws the selected edge with `border-color`,
not `outline`; and because the two levels' values are baked into the emitter, no theme could rebind
them. The carrier grammar plus R-STATE-10's prefix already solve the general shape — compose
carrier words, scope them to a condition — so selection needs nothing new except the one thing that
separates it from hover: it is asserted by the application, not the platform, and must be backed. A
`selected:` prefix reaching an element the container never marked selectable is a bug, so the
linter verifies the `selectable` capability and the container's assertion (R-STATE-08) before the
override is legal; the CSS serializes to the same attribute selector the backing already implies
(`[aria-selected="true"]`). This unifies selection with hover under one prefix mechanism and lets a
project's actual selected look compose from carriers, retiring the fixed subtle/strong levels that
could express neither. `checked` is its sibling (`[aria-checked]`). Surfaced when the selection-sink
wiring hit the hue/property mismatch. Source: ADR-0008.

## RAT:R-STATE-12
Monky's modal navigation marks the active tab with `aria-current="page"` and colours it locally —
three declarations (accent ink, subtle ground, accent bottom rule) that every reconciliation pass
since U5 deliberately refused to recast as `selected:`, because `aria-current` is a different
assertion with different semantics: currency within a set (navigation, steps, pagination), not
selection. The carriers and the prefix mechanism already fit; what was missing was a backing
story. `selected:`/`checked:` verify a capability word because their state is distributed — a
container asserts it onto children, and a prefix without the `selectable` contract is a bug the
linter can catch from the class string alone. `aria-current` has no container contract to verify:
the element itself carries the attribute, so the only honest backing is the attribute, and the
serialization enforces it — `[aria-current]:not([aria-current="false"])` cannot match an element
the application never asserted. The `:not` guard exists because `aria-current="false"` is a legal
way to write "not current"; matching it would fire the override on an explicit negative. A
capability word (`navigable`) was weighed and set aside: it would verify nothing real, since no
distributed container state exists to entail, and inventing one to satisfy the R-STATE-11 shape
puts ceremony ahead of truth. Surfaced by the aria-current rows held back across three
assimilation pilots. Source: ADR-0009.

## RAT:R-SKIN-08
The theme-plane sessions plus the W3C Design Tokens / Style Dictionary model: a socket is a
semantic token resolved by theme × mode. The socket list must be registry-owned so the plane
has an identity a project cannot fork; a Radix-style ramp may generate values under it.
See `docs/skin-theme-ruling-draft.md`.

## RAT:R-SKIN-02
The rectangle-delimiter derivation (`docs/SKIN-GRAMMAR-PROPOSAL.md`): the semantic choice is
whether the rectangle is made perceivable; the means (tone / border / radius / shadow) are
theme facets of one delimiter, not separate axes. Flush is the unmarked default, mirroring an
SVG shape with `fill="none" stroke="none"`.

## RAT:R-SKIN-03
The carrier/role/intensity model, independently confirmed by GitHub Primer's
`[carrier]-[role]-[intensity]` (fgColor / bgColor / borderColor = ink / ground / rule). A role
rides a carrier so it never collides with a property owner. See `docs/skin-theme-ruling-draft.md` §5.

## RAT:R-SKIN-04
Anchor-plus-recession from the ink/ground games. Four steps kept for expressive range (a
current-trend observation) against Primer's leaner default/muted/subtle; provisional with a
data-triggered reduction. Realization deferred to the theme so alpha vs `color-mix` stays an
implementation choice. Refinement (U8e, Monky evidence): realization includes the theme
*binding a step's value directly* — hand-tuned or computed — not only a formula. The socket
contract names each step; a theme supplies the value or leaves the (optional) step unbound.
Ground varies toward differentiation rather than recession, so it names its own steps
(`subtle`/`defined`) instead of reusing the recession words.

## RAT:R-SKIN-05
Roles as a shared reporting register (pass/warn/fail/note) that read as siblings without one
axis; validated against Primer's roles (accent/success/attention/danger). `note` kept distinct
provisionally though Primer folds neutral info into accent. The constrained interface palette
versus the versatile data plane resolves the restrict-vs-forgive tension by design-for-the-
regularity: target the regular case, push the exception (dashboards/graphs) to a separate plane.

## RAT:R-SKIN-06
Corner kind from SVG `stroke-linejoin` (miter/round/bevel); magnitude scale-bound with fixed
endpoints `square` and `pill` (pill = `calc(infinity * 1px)`, clamped by the spec to half the
shorter side).

## RAT:R-SKIN-07
`font` is the delimiter-pattern kind (a multi-property responsibility area), not the ink-pattern
kind (single property). The apparent asymmetry with ink dissolves at the description level:
words name intent, not CSS properties (the R-SKIN-01 seam). Namespace locality keeps all
typography under `font-*`.

## RAT:R-SKIN-09
The `shadow` colour socket shipped with the theme plane, but every shadow *geometry* stayed a
project literal — the Monky current ledger carries five elevation rows (modal dialog, style
dropdown, editor toast, suggestions container, command suggestions), a named follow-up since the
theme binding. Elevation is a treatment, not a carrier: a carrier is a colour riding one property,
while a cast shadow is a composite value whose colour is themed and whose geometry is a number
choice — exactly the grammar-owns-the-word / theme-owns-the-numbers seam (R-SCALE-03). The socket
carries a default geometry the way carriers carry default colours (R-SKIN-03), so `elevated`
works before a theme binds `--shadow-elevated`. Only `elevated` is admitted: the dropdown-tier
evidence is uniform, but the modal's two-layer multiply-blend shadow is an identity signature, and
the one inset occurrence is an error ring, not depth — so `recessed` stays reserved, the
hover/focus admission pattern. Naming: the follow-up's working names were `raised`/`sunken`, but
`raised` is already a z-scale stacking tier and one word resolves to one axis, so the treatment
takes `elevated`/`recessed`; size-graded names (`shadow-sm/md`) were set aside because a
treatment names what the surface does — sits above or below its ground — not how many pixels of
blur that takes, the what-not-how bar. Source: ADR-0010.

## RAT:R-SKIN-10
Every assimilation pilot since the colour pass hit the same wall and named it: `.settings-view
.btn-outlined` "is blocked by the shared `.btn-outlined` skin rule; changing it requires a button
recipe decision". The Monky evidence is the `.btn` family (base, secondary, outlined, success,
link, link-danger, disabled, active) plus `.alert-*`, `.card`, and `.input` — bundles that
repeat across surfaces as one decision. Two resolutions were weighed and rejected. Admitting
recipe words (`btn`, `alert`) fails the admission test: the word names a component kind, and its
content — which carriers, which radius, what disabled looks like — is project identity that no
two projects share. Dissolving recipes into markup composition fails the other way: a recipe is
one decision made in many places, and repeating six words per button is repetition masquerading
as flexibility, while the recipe's conditional identity (disabled, active, context overrides like
the settings-view competition) still needs a stylesheet home. The boundary is the resolution: the
grammar rules the planes recipes are built from, the project owns the bundle, and the discipline
that keeps the seam honest is socket consumption — a recipe class may not contain a colour
literal (Monky enforces this with its Phase B stylelint guard). This gives the recipe rows a
terminal citation without inventing vocabulary the admission test rejects. Source: ADR-0011.

## RAT:R-SKIN-11
The largest single family in the Monky residue screen: ~49 declarations that say nothing but
"this element has an edge line" — `border-width: 1px; border-style: solid`, all-sides or
per-side, on every separator, container edge, and keycap. The colour half was solved by the
`rule` carrier; every pilot then split shorthands by hand (width/style local, colour to the
carrier) and the width/style halves accumulated as mechanics with no words. The presence word
completes the split. `ruled` was chosen because it pairs with the carrier it composes with —
`rule ruled` reads as one thing — over `edged`/`bordered`, which name the CSS property rather
than the design intent and lose the carrier tie. Physical sides follow the evidence (separators
are visually top/bottom lines); logical-side forms remain open for when writing-mode evidence
exists. Weight is a socket with an Ermine default, not a scale: the evidence shows exactly one
step (1px everywhere; the one 2px is a selection-indicator underline, which is identity, not a
rule line), and R-SCALE discipline does not invent steps without measured need. With both the
shadow (R-SKIN-09) and line halves ruled, the sampled skin-surface gap axis is fully resolved
and retires. Source: ADR-0012.

## RAT:R-SKIN-12
Five selectors across four surfaces carry the identical trio — `overflow: hidden;
text-overflow: ellipsis; white-space: nowrap` — and Monky had already half-named the intent
for itself (`.popup-truncate`). The trio is one intent spread over three properties in two
responsibility areas, so no carrier can own it and no lawful composition of pre-existing
words could express it: a treatment, the R-SKIN-09 shape. It deliberately does not own
overflow. The overflow axis owns `overflow-x/y` (P7), and folding a third property into the
treatment would either collide or demand a new sanctioned-share mechanism; instead the
treatment composes with the newly admitted `hidden` word — `hidden truncate` — the same
explicit seam as `rule ruled`, where the markup states both facts and ownership stays
disjoint. The search view's release of truncation under `aria-selected` is evidence the
treatment wants a conditioned form eventually; today the release stays a local override.
`truncate-N` (the line-clamp form) has one occurrence and stays reserved — the
tween/choreography admission pattern. Source: ADR-0013.

## RAT:R-SKIN-13
The pattern screen found the focus indicator was never scattered mechanics: seven
`outline: none` suppressions pairing seven `box-shadow: 0 0 0 2px var(--tone)` draws, one
treatment implemented per-component. The pairing is also the hazard — one suppression already
rides without its ring, which is exactly the accessibility failure mode the suppress-and-redraw
discipline invites. RAT:R-STATE-10 called redrawing the UA indicator the anti-pattern; the
resolution is not to bless the redraw but to dissolve it: `ring` restyles the platform's own
`outline`, so the indicator never leaves the platform's hands, the colour and weight come from
the theme (`--ring` socket, default on the `ground-defined` tone the evidence uses), and there
is no suppression left to drift. `outline` was verified unowned; `box-shadow` stays with
elevation, which is why box-shadow rings are explicitly not `ring`. The authored form is
`focus:ring` — the treatment has no meaning outside a condition, and the existing R-STATE-10
prefix supplies it; serialization to `:focus-visible` instead of `:focus` is the anticipated
refinement once the platform-condition family admits it (today's evidence is all text inputs,
where the two match identically). The error-tinted inset ring on `.input-error:focus` is recipe
conditional identity and stays in the recipe. Source: ADR-0014.

## RAT:R-SKIN-14
The skin-type gap axis held two properties. They resolved oppositely, which is the point of
evidence-gated admission. Text alignment has real vocabulary: four `center`s (chips and empty
states) and two `left`s that are both the same intent — a button restoring natural alignment
against the platform's centring — so the words are logical (`text-start`, `text-center`),
platform-first, with `text-end` reserved. Line-height has none: the 1.5 base is already
substrate (reset and entry-point font), the one component-sheet copy is that substrate
duplicated into a shadow root, `normal` on the editor body is user-content neutralization, and
the single `line-height: 1` is the keycap identity signature. A leading scale invented from
one deviation would violate R-SCALE discipline, so leading stays substrate/theme-owned with
identity deviations local. Two single-occurrence candidates are noted for future evidence, not
admitted: `font-variant-numeric: tabular-nums` (a natural `font-*` facet if numbers recur) and
the uppercase/letter-spaced label treatment. With alignment ruled and leading dispositioned,
skin-type retires — the registry's last gap axis. Source: ADR-0016.

## RAT:R-SKIN-15
Scrollbars were the adoption's longest-standing follow-up: 24 frozen-ledger rows, 19 live ones,
and two competing implementations in one codebase — a global engine-drawn treatment
(`::-webkit-scrollbar*` at 12px with `!important`) and the popup's standard-property thin form,
which Phase B had already moved onto sockets. The ruling follows the platform seam the evidence
exposes. The standard properties (`scrollbar-width`, `scrollbar-color`) are what both target
engines support and what the platform itself prefers when both forms are present, so prominence
is ruled there: `scrollbar-subtle` = thin, thumb/track from like-named sockets, defaulting to
the `rule` carrier over a transparent track — a scrollbar coloured like the project's lines.
The engine-drawn form cannot be a treatment without inventing multi-selector pseudo-element
emission for one word, and its 12px/radius/hover geometry is a look, not a prominence — it
stays identity, exactly as the modal's blend-mode shadow did under R-SKIN-09. `scrollbar-hidden`
(carousels, kiosk surfaces) is anticipated but unevidenced, so reserved. Naming rides the
existing intensity vocabulary (`subtle`) rather than coining size words. Source: ADR-0017.

## RAT:R-STATE-13
The reveal-on-row-state pattern appeared independently in two surfaces (the search row's edit
icon, the suggestion row's delete action) — the same recurrence bar that admitted `hover:` —
plus row-tinting variants, fourteen rows in all. Same-element prefixes cannot express any of it.
The mechanism question was the hard part: a naive relational serialization (`*:hover .word`) is
wrong by construction, because hover propagates through the whole ancestor chain — hovering
anywhere in the modal would reveal every icon. The bound comes from the same place selection's
backing came from: R-STATE-08's distributed contract. Both evidence ancestors are selectable
rows, so the relational prefixes anchor on the `selectable` capability — the ancestor that
declares itself the interactive unit is the one whose state speaks — and the linter verifies
the pair through parent context (the P11 pattern: verify when context is given, skip when it
is not, and the serialization remains safe either way because an unmarked ancestor never
matches). Narrow admission: `parent-hover:` and `parent-selected:` only; the guarded tinting
rows (`:not([data-state])` refinements) stay local as component mechanics rather than
stretching the prefix to carry guards. Source: ADR-0018.

## RAT:R-SKIN-16
The relational prefixes alone could not consume their own evidence: the revealed property is
opacity, which had no word to scope. The endpoints earn words — `concealed`/`revealed` name
presence (a thing that occupies space but does not show), which is a real platform distinction
(opacity 0 keeps layout, hit-testing, and measurement; `display: none` and `visibility` do
not). The mid-scale does not: 0.35 marks, 0.6 carets, and 0.95 disabled washes each name a
different emphasis intent, one occurrence apiece — admitting a scale there would repeat the
line-height mistake R-SKIN-14 refused. Pointer-events is deliberately left out of `concealed`:
the evidence keeps concealed controls clickable, and folding interactivity into a visibility
word would smuggle behaviour into skin. Source: ADR-0018.

## 0. What this document is, and how to read it

**Goal.** This is the **source of truth** for a small *semantic style grammar* — a vocabulary of
perceptual/relational words composed in markup so that the meaning of a UI survives without reading
the CSS. Its job is to **legislate the grammar once, completely, and consistently**, so that
everything else (the CSS that ships, the guide humans read, the spec a machine validates against) can
be *derived* from it rather than re-decided. It settles *what the axes are, how they compose, and what
makes a class string well-formed* — and nothing about a specific visual style, which is deliberately
left to skin (§5) and to measurement.

**It is a design-phase artifact, not a manual.** This document records a *design process still in
motion*: it carries its own history (retired ideas, corrections, the reasoning behind each ruling) on
purpose, because the reasoning is what lets a future reader — or a later version of this work — re-derive
a decision instead of merely obeying it. Markers tell you how settled each part is:
- **(settled)** / **RULED** / **RESOLVED** — load-bearing; do not relitigate without cause.
- **`[RULING]`** — a remaining decision; by this stage, only *value/name* choices, never structural ones.
- **EXTRACTION-STABLE / FROZEN (slot)** — fixed enough to extract from; see the status block. Note
  the deliberate scoping: *extraction-stable* ≠ *structurally final* — a future component audit could
  still force a structural revision (a new sibling grammar, a re-cut role). Empirical validation across
  many component families is ongoing.
Read it as a constitution with a paper trail, not as a finished reference card.

**Its register is precise, not friendly — by design.** This is a **typed, controlled grammar over
CSS** (§2 honesty note): it renames CSS/ARIA/layout-engine concepts into a lawful, checkable
vocabulary. It is *not* "natural-language CSS," and it is not written for first-time authors. The
friendly, example-led version is the derived **human guide**; the terse, machine-validatable version is
the derived **spec**. This source is the place where the rules are *argued and fixed*, so it reads like
law, not like onboarding.

**The three-document system.** This constitution is the **single upstream source**; two artifacts are
*extracted* from it and must never drift from it:

| Document | Audience | Register | Relationship to this source |
|---|---|---|---|
| **ERMINE** (this) | the author / maintainer | constitutional, with rationale & history | the source of truth; the only place rulings are *made* |
| **Well-formedness spec** (derived) | LLMs, linters, validators, tools | register-flat, predicate-form, no metaphors | a mechanical extraction: the axis registry + the laws as checkable predicates |
| **Human guide** (derived) | authors using the grammar | friendly, example-led, task-oriented | a teaching extraction: the common cases, worked examples, the small public surface |

The split exists because the three audiences want **incompatible registers**: the rationale and
metaphors that help a human *understand* the design actively *hurt* machine parsing, and the terse
predicates a validator needs *bore* a human learner. Forcing all three into one document was the thing
deliberately avoided — but only by holding everything here *until the structure stopped moving*, so the
extractions are clean (no lateral edits propagating across three files). The **machine-readable axis
registry** (§2) is the literal pivot: the spec is its predicate face, the guide is its example face, the
shipped CSS is its generated face. Change a rule *here and in the registry*; regenerate the rest.

**How to use this source, by role.** *Extracting the spec:* pull the axis tables (name · role ·
signature · vocabulary · members · default · `must-never-touch`), the §1–§2 laws as predicates, the §1

**The shape of the system (orientation).** One **constitution** of shared laws (planes · roles ·
scope, §§1–3) governs the **grammar plane's four sibling grammars** — **layout** (in-flow), **state**,
**motion**, **layering** (out-of-flow) — with **skin** and **identity** as the other two planes.
NB: the *umbrella* is the **style** grammar, not "layout" — "layout" now names one sibling
(in-flow arrangement), held distinct from "layering" (out-of-flow depth). A first taste:

```html
<!-- grammar words (behavior) + per-property spacing over the shared scale -->
class="horizontal gap-comfortable padding-relaxed"
```

The words are grammar (how it behaves), not skin (how it looks): `horizontal` is the container's
structure; `gap-comfortable` / `padding-relaxed` are per-property spacing over one shared scale (§6).
Each names a distinct axis, so they compose without conflict. (Note this is a *typed* surface, not
"natural-language CSS" — see the honesty note in §2: the words rename CSS concepts into a controlled,
checkable grammar; an author still reasons in layout terms, just lawful ones. And `selected`-style
**state** is shown separately below, because state is a *condition*, not a spacing/skin bundle — §4.2.)

**What Ermine is — and what it is *for* (present vs direction).** Two registers, kept apart so the
second never inflates the first.

*Present (demonstrable, now).* Ermine is a **framework**: a typed, composable style grammar with an
executable registry and a linter. What lifts it past a utility set is the seam it enforces — **classes
name meaning, not values** (`gap-comfortable`, not `gap-4`), and their composition is **checked**
(disjoint properties, one-word-per-axis, entailment); the visual values are isolated behind a `var()`
seam a swappable theme owns (§5–§6). The whole present claim is bounded and testable: *a semantic,
composable framework whose class strings are verifiably well-formed and whose look is isolated behind a
theme seam.* Nothing in it requires anything that does not already run. (This is the **competence**
grammar of the status note; generating and predicting *into* it is **performance**, below.)

*Direction (GOALS — non-normative).* Because the grammar is a **finite, typed language with composition
laws**, well-formedness is a constraint an author — human, tool, or model — can be held to. Two goals
follow, in the same direction, at increasing distance and decreasing certainty:
- **(near goal) generation *into* the grammar** — the linter's well-formedness rules *are* a generation
  grammar; a model given the spec's authoring contract can be constrained to emit only valid Ermine. A
  goal, not a feature: no generator ships today.
- **(far goal) a layout model** — given a corpus of real designs *parsed into* the grammar (the inverse
  of the emitter — unbuilt, and the hard direction; see §5.1), characterize the distribution of real
  layouts and generate *statistically* plausible ones. The north star, not a claim: it rests on
  artifacts that do not yet exist.

The boundary between the two registers is load-bearing: **validity is grammatical** — the grammar
delivers it now; **quality is statistical** — it needs the corpus, later. The framework is present
tense; the layout model is a distant goal *in the same direction*, written here as a goal precisely so
it is never read as a capability.

## 1. The three planes (settled)

Sort every style by **function**, not by file or by reusable-vs-app.

| Plane | Question | Nature |
|---|---|---|
| **Grammar** (4 siblings: layout · state · motion · layering) | *How does it behave/arrange?* | reusable, perceptual, app-agnostic |
| **Skin** | *How does it look?* | partly lawful, partly learned (see §5) |
| **Identity** | *What specific thing is it?* | one-off, app-bound; composes the other two |

**Decomposition test:** a class that sets layout (`display`/`gap`/`align`) *and*
appearance (`background`/`border`/`color`) mixes planes and **must be split**.

**Goal for identity:** an app component should *compose* grammar + skin in markup and
shrink to its true residue (selectors, one-off positioning), not re-implement them.

### Scope — radius of action (settled — Jorge's ruling)
Orthogonal to plane and role: every rule has a **radius of action**. It is NOT a fixed ladder
— it's a **nesting of scopes mirroring the DOM tree** = lexical scoping for style (resolution
walks up the enclosing scopes, **nearest wins — within one ranking metric**, see below). Narrow → wide:
- **element** — the element's own singularities (overrides; innermost).
- **container** — governs **direct children** only (`> *`). Natural range of structure/
  member-role. (Depth: one level, *not* subtree.)
- **ambient — local or global** — a default set at a scope ROOT and inherited by its subtree
  until shadowed. **Recursive:** any region can establish its own ambient (a card's house
  style); nested ambients override outer ones; the **global** ambient is just the outermost
  case. This is the **cultural baseline** → skin's unmarked default (§5), and it may also carry
  *grammar*-defaults for a region (e.g. a denser zone). Scope is a **mechanism, not a plane** —
  what it carries stays plane-tagged.
- **substrate / reset** — neutral ground: normalize variance, establish a known zero
  (`* { box-sizing }`, form-control `font: inherit`). *Removes* variance, *adds* no style →
  **not cultural**; plane-agnostic 0th layer.

#### Resolution law — membership, then one ranking metric (settled)
"Nearest wins" is only well-defined once you fix **what "nearest" measures** — and there are
**three different distance metrics** in play, not one. Treating them as interchangeable is what
made the ladder feel coherent in prose but underspecified in fact: tree-proximity, re-declaration
depth, and boundary-membership are *different functions*, and each defines a different "nearest."
The law that restores a single well-defined winner per property runs in **two stages**:

**Stage 1 — membership (set, not rank).** A hard boundary (**Shadow DOM `:host`**) decides *who is
even allowed to compete* for a property on an element — it is a **set operation** (excludes
outsiders entirely), run **before** any ranking, not a ranking itself. Treating `:host` as one
more "scope level" was the category error: it does not rank claimants, it removes them from the
contest. Use it for exactly the hard-isolation case §4.4 needs — the overlay-vs-host-page
quarantine (the overlays ALREADY are membership boundaries: suggestions vs modal `:host` fonts).

**Stage 2 — ranking (one metric per property).** *Within* a membership set, a property is governed
by **exactly one** ranking metric. Two metrics must never rank the same property — that is the
condition under which "nearest wins" is single-valued:
- **open-ambient** (custom-property inheritance) — metric = **re-declaration depth**; *unbounded*,
  flows down the subtree until shadowed. The default mechanism for the **ambient** ladder rung
  above: the cultural baseline and region grammar-defaults.
- **bounded-regional** (`@scope (.root) [to (.boundary)]`) — metric = **tree-proximity with an
  explicit lower edge** (the *donut*); for a default that must **die at a nested boundary** short
  of a Shadow root. **Conformance clause, not a shipped mechanism (see below).**

The **bounded-vs-open** choice is real, but it is a choice of *which job*, not a free pick among
equals: if the default should stop at an edge → bounded-regional; if it should flow until
shadowed → open-ambient. Never both on one property.

> **`@scope` — legislated, not implemented (conformance clause).** This grammar does not currently
> ship `@scope`; nothing here uses the donut. It is recorded as the **conforming mechanism for the
> bounded-regional job** so a third-party consumer who *does* need a default that dies at a nested
> boundary finds the slot already defined and the law already covering them. It occupies the
> bounded-regional ranking metric and **must not** also govern a property governed by open-ambient
> inheritance in the same subtree. An edge case earns a *clause in the law* as soon as it is
> conceivable for a conforming user; it earns a *mechanism in the codebase* only when a failing
> test demands it (cf. the retired value-channel, §6.3). `@scope` is in the first bucket.

The "Universal Base" block in layout-semantic.css mixes substrate (box-sizing) + global ambient
(app font) + skin (scrollbar) → separate by layer per the decomposition test.

## 2. The grammar's laws (settled)

**What this grammar is (honesty note).** This is a **typed, controlled grammar over CSS** — not
"natural-language CSS." Its words *rename* CSS/ARIA/layout-engine concepts into a lawful, checkable
vocabulary; an author still reasons in layout terms (basis, grow, negotiation), just lawful ones.
`elastic basis-ratio grow-2 self-center` is more *semantic* and more *composable* than `flex: 1 1 0`,
but it is not "simpler for someone who doesn't know flexbox." The grammar's promise is
**meaning-preservation + compositional law + machine-checkability**, not the elimination of CSS
knowledge. Claims of "natural language" would overstate it; "typed utility grammar" is the honest
description.

**Two composition regimes (settled — the keystone).** Not every axis composes the same way,
and forcing them to was the source of the apparent algebra-breakage at sizing. The grammar has
**two regimes**, and each law below states which it governs:
- **Free composition** — independent axes that *commute and unify*: declaring one says nothing
  about another, and the rendered effect of the pair is the union of the parts. Governs
  **placement, structure, state, motion, layering, skin** — the bulk of the grammar.
- **Negotiated composition** — a per-member *demand* resolved by a **global solve** over the
  parent's finite space. The members do **not** commute; the result is a joint settlement, not a
  union. Governs exactly **one** axis-family: member-role sizing (give↔grab + basis + the
  min/max band). This is not a defect to be designed away — it is the irreducible character of
  laying out **dynamic content under finite space**, and it is the one place the grammar runs a
  solver instead of a free product.

The two regimes do not fuse; they **sequence** — `place ∘ negotiate` (Law 8). The solver settles
sizes globally **first**; placement positions each settled box locally and compositionally
**after**. Everything entangled (the §6.4 freeze-loop, min/max clamping) lives *inside*
`negotiate`; nothing in `place` ever touches it. So the clean separation the decomposition test
demands is recovered — it runs between *phases*, not between *axes*.

1. **Axis = paradigm.** Each axis is a choice-set; a word names one choice in it.
2. **One word per paradigm (unification).** A well-formed class string picks *at most one*
   word per axis **per condition scope**. Two words from the same axis *in the same scope* is a
   conflict (`horizontal vertical`), not a composition — the join does not exist. *(Both regimes.)*
   **Condition-scope refinement (§4.2 environmental ruling):** environmental-state variant prefixes
   (`viewport-md:`, `container-lg:`, `prefers-*:`) open a *new* scope, so the same axis may recur once
   per scope — `horizontal viewport-md:vertical` is well-formed (base vs `viewport-md`), while
   `viewport-md:horizontal viewport-md:vertical` (same scope) conflicts. Bare interaction/input states
   do **not** open scopes. The linter buckets by scope, then applies one-word-per-axis within each.
3. **Dimensional purity = the enabler of composition** (not a restriction) — *for the free
   regime.* A word touches *only* its own axis's properties — and this single responsibility is
   precisely *why* free-regime words compose: because no two axes touch the same property, classes
   from different axes **commute and unify without conflict** (Law 2). Purity isn't a prohibition
   on a class; it's the guarantee that makes the *free* algebra work — exactly as referential
   transparency ("no side effects") is stated as a constraint but *is valued* for composability.
   The per-axis **`must-never-touch`** field is the operational, checkable form of this enabler
   (what an audit/linter tests), not the principle itself.
   **The negotiated axis does not obey this law** — its members deliberately contend over one
   shared resource (the parent's space), so they do *not* commute. Purity is replaced there by the
   solver's own laws (see the `negotiated-field` signature).
4. **Default = markedness.** Each axis declares an unmarked default: what a bare element
   means with no word from that axis. Defaults are part of the grammar.
   **Defaults ride the cascade:** a container may supply the normalized default for its
   members; an element overrides locally (its singularity). Distribution must be
   **zero-specificity** (`:where(.blocks > *)`) so the element-level override always wins.
5. **Contrast (value through difference).** A word earns its place only by differing from
   its neighbors. Two words with the same effect = one word with two spellings → delete one.
6. **Name the platform's distinctions; mint no new ontology.** A word is a *candidate* only if it
   names a distinction the platform **already makes** — in CSS (the box model, outer/inner display,
   `position` modes), in ARIA / DOM (interaction states), in the layout engine (give↔grab, basis).
   The grammar's job is to give these existing distinctions *perceptual, composable names*, never to
   author distinctions the platform does not have. This is the positive criterion behind the
   no-coining meta-rule: no-coining is the *prohibition* ("don't invent words"); this is the *test*
   that says whether a proposed word is even eligible — **does the platform draw this line?** If not,
   it is not a missing word but a **non-distinction**, and the felt "gap" is usually either a
   composition across existing axes or a skin/identity concern, not a new grammar member. This law is
   why so many candidates were already retired — `greedy` (a surplus *symptom*, not a platform
   distinction), `spaced` (a synonym), `sections` (HTML landmarks already name it), `equal-circle`
   (size + radius, two platform distinctions fused) — each failed *this* test, not merely Contrast.
   For the **state** sibling (§4.2) this law is load-bearing and literal: states are *mirrored* from
   ARIA / platform, never coined.

   **Law 6b — mirroring is rebuttable by Contrast (added this revision).** A platform distinction is
   *eligible* under Law 6 but not *automatically admitted*. If two platform-named states produce no
   difference the grammar can act on — same effects, same composition, same entailment surface — they
   **fail Contrast (Law 5) and collapse to one grammar member carrying the richer backing as an
   entailment *set***, not two members. Mirroring imports the platform's *distinctions*, not its
   *redundancies* — ARIA and CSS are accreted, not designed, and some of their lines are committee
   artifacts, not real perceptual distinctions. **Guardrail:** Law 6b can only ever *reduce* member
   count, *never increase* it, and *never coins* — it picks the better of two existing platform names
   and folds the other into the entailment set. It is strictly weaker than coining: a merge, not a
   mint. Worked case: ARIA's `aria-pressed` (toggle) and `aria-selected` (set-item) produce the same
   effect surface in this grammar → Contrast collapses them to one member (`selected`) whose entailment
   is the disjunction `selected ⟹ aria-selected ∨ aria-pressed ∨ :checked`, role-picked. Consequence
   for the schema: `state-instance` carries a backing **set** (any-one satisfies), not a backing atom.
7. **Compositionality — *for the free regime*.** There, the meaning of the whole = a function of
   the parts + these rules, computable from the parts in isolation. **The negotiated regime is
   not part-wise compositional** — a member's settled size is a function of *all* siblings and the
   container, not of the member alone — and does not claim to be; its honesty comes instead from
   **outcome-determinism** (same demands + same container → same result, every time) and from the
   solver laws being *testable invariants* (§10). The contract that keeps the human-readable
   surface and the machine-checkable engine in agreement holds in **both** regimes — it is just
   *part-wise* in the free regime and *invariant-wise* in the negotiated one.
8. **Phase order (negotiate-then-place) — settled.** When an element carries both negotiated
   demand (sizing) and free placement (alignment, grid placement), the effect is **`place ∘
   negotiate`, never the reverse**: the solver fixes the box, then placement positions it. The two
   never interleave. This is what lets a single element legitimately carry `expandable` *and*
   `self-center` at once (the *container-and-member* composition, below) without the entanglement
   of one leaking into the other.

### Algebraic signature (settled as a *requirement*; values per-axis below)
Every axis declares **one** signature, which determines how it composes and what to test:
- **set-with-exclusivity** — unordered choices, mutually exclusive (e.g. structure). *Free.*
- **ordered-chain (scale)** — a totally ordered scale (e.g. density). *Free.*
- **container-operation** — an n-ary operation over children; the recursion of the DOM
  tree lives here, not in the class string (e.g. `horizontal` as a layout operator). *Free.*
- **negotiated-field** — a per-member *demand* (`want` · `give` · `grab`, + a self/internal
  `bounds` clamp) resolved by a **global solve** over the parent's space. **Does not satisfy
  Law 3.** Its laws are the §10 invariants, which *are* its algebra: **conservation**
  (Σ settled = available space), **monotonicity** (more `grab` never shrinks you; more `give`
  never grows you), **ratio-invariance** (scaling all weights by *c* leaves the result fixed),
  **order-equivariance** (permuting members permutes the result identically), and
  **clamp-idempotence** (applying `bounds` twice = once). The solver is the oracle (§6.4); these
  invariants are what the grammar legislates and tests, not a hand-derived formula.

### Every axis also declares its vocabulary — closed or open (settled — the LLM-conformance field)
Orthogonal to signature: each axis states whether its member set is **closed** (the words are
exhaustive — *do not coin new ones*; a missing distinction is either expressible by composition
or a gap to report) or **open / parametric** (admits a numeric or generated parameter by a stated
rule — e.g. `grow-N`, the only axis whose *whole substrate* is open). Closed is the default and the safe case. This field
exists for the same reason `must-never-touch` does: it is the machine-checkable guard an automated
consumer (or linter) needs — the dominant failure mode when *extending* the grammar is **inventing
a plausible-sounding word**, which the alias law (§4.1) already forbids and this field makes
checkable per-axis.

#### Two primitives, applied at two scopes (settled — the vocabulary-shape revision)
Vocabulary has exactly **two primitives**: `closed` (an exhaustive word set) and `open` (a parameter
admitted by a stated rule). There is **no third primitive.** What looked, across several axes, like
richer vocabulary kinds turned out to be the *same two primitives applied at a different scope, or
carrying earned sugar*:

- **`open` at axis scope, with whole-axis aliases** — **m2 (flex).** The axis is genuinely open: two
  independent dials, `grow-N` and `shrink-N`. The four corners (`rigid`/`compressible`/`expandable`/
  `elastic`) are **whole-axis aliases** — each names a *complete* value (it fixes *both* dials at once,
  e.g. `elastic` = `grow-1 shrink-1`). Because an alias is a complete value, it is **mutually exclusive
  with every other word on the axis**: `expandable shrink-2` and `rigid grow-2` are conflicts, not
  refinements. You write *either* an alias *or* the numbered dials, never both. Numbered dials compose
  one-per-dial (`grow-2 shrink-1`). The aliases are sugar earned by recurrence (the alias law), not a
  second vocabulary.
- **`closed` at axis scope, with a parametric member** — **m3 (basis), m5 (span), and the enumerated
  states (`sorted`, `current`).** Membership is *closed* (you cannot coin a new member), but one member
  is **parametric** — it carries a value: `basis-exact-<size>`, `span-<N>`, `aria-sort` ∈ {…}. Note the
  parameter *itself* may be closed or open: `basis-exact` is **closed** over the size-token set (no
  arbitrary lengths), `aria-sort` is a **closed** ARIA enum, and only `span-<N>` is genuinely **open**
  (unbounded integer). So the precise statement is *closed axis, one member parametric; the member's
  parameter is closed or open* — `open` applied at member scope is one case of it, not the whole of it.
  This is exactly the mechanism the enumerated states already use. It is **not** an open *axis*:
  `basis-content basis-exact-md` is two members of one closed axis, a conflict. m5's `span-all` is a
  *contextual* member (it resolves to the grid's column count, like `grid-column: 1 / -1`), not a value
  of `span-N` — another closed member, not an axis-scope opening.

The discipline is Law 6 turned inward on the grammar's own vocabulary: **mint no new ontology.** A
"new shape" that decomposes into (primitive at a scope) + (optional earned alias) is a *composition*,
not a primitive, and is admitted as such — never as a third vocabulary kind. The tell that a candidate
is real rather than illusory: it cannot be expressed by the two primitives at any scope. None of the
sizing/placement shapes met that bar; all reduced — so **no third primitive has survived examination**
(stated as the evidence supports: three candidates examined and reduced, not a proof of impossibility).

### Meta-rule — no coining (settled — the extension contract)
The dominant failure when *extending* this grammar — especially by an automated consumer — is
**inventing a plausible-sounding word** (`stretchy`, `loose-wide`, `centered-grow`). The grammar
forbids it, and the prohibition is checkable:
1. **Closed axes admit no new members.** If a distinction seems missing on a closed axis, it is
   either already expressible by *composition* across axes, or it is a genuine **gap to report**
   (a `[RULING]`), never a word to mint. A consumer that wants a word not in a closed axis must
   stop and surface the gap, not guess.
2. **Open axes admit only their stated parameter**, by the axis's rule (`grow-N`, `span-N`,
   `exact-N`, a duration) — not free-form new *words*, only new *values* of the sanctioned
   parameter.
3. **Aliases are earned, never invented** — the alias law (§4.1): a shorthand is codified only
   after a primitive combination demonstrably recurs in real use. This is the *one* sanctioned
   route by which the closed vocabulary ever grows, and it runs from observed usage inward, never
   from speculation outward.

Together with `must-never-touch` (per-axis property scope) and `vocabulary` (per-axis open/closed),
this completes the machine-checkable surface: an automated consumer can validate a class string —
and reject an over-eager extension of its own — using only string-and-axis logic, with no knowledge
of what any CSS property *does*.

**The product is the registry, not the CSS file.** These laws (one-word-per-axis, closed/open
vocabulary, no-coining, `must-never-touch`, state⟹ARIA entailment, `rigid grow-2` invalid,
`horizontal vertical` invalid, grammar-must-not-touch-skin) are **documentation only** until something
enforces them. The real deliverable is therefore a **machine-readable axis registry** — one record per
axis (`{axis, role, signature, vocabulary, members, controls, mustNeverTouch, regime}`) — that is the
**single source of truth**, from which the CSS classes, the documentation, the lint rules, the
invalid-combination tests, and migration suggestions are all *generated*. The constitution is the
*specification* of that registry; the registry is what makes the constitution operational. (This is
the first derived artifact to extract — see the extraction plan; the spec/well-formedness document is
the human-readable face of the same registry.)

### Two roles every element plays (settled — Jorge's ruling)
Every element is simultaneously a **container** (toward its children) and a **member**
(toward its parent). These directions are **orthogonal** — they never conflict, always
compose (an element can be `horizontal` *and* `fill-remaining` at once) — so each axis
must declare which role it speaks to:
- **container-role** ("looks down") — governs how *children* arrange/space/align.
- **member-role** ("looks up") — governs how *this element* sizes/flexes/aligns *itself*
  within its parent; it acquires properties for itself, it does **not** impose on children.

`structure` is container-role. blocks/items + sizing are **member-role** (not structure).
The cut also splits later axes: alignment = `align-self` (member) vs `align-items`/`justify`
(container); `constraints` (min/max) = member-role.

**Role and regime are orthogonal.** Member-role contains *both* regimes: placement sub-axes
(m4 self-align, m5 grid placement) are **free**; the sizing sub-axes (m2 give↔grab, m3 basis,
the min/max band) are **negotiated**. The container/member cut answers *which direction a rule
looks*; the free/negotiated cut answers *whether it commutes or contends*. Keep them separate —
collapsing them is what made the member-role family read as "five sub-axes that compose without
conflict," which was never quite true: four of them compose freely, the sizing ones negotiate.

## 3. Axis entry template

```
### <axis name>
- signature: set-with-exclusivity | ordered-chain | container-operation | negotiated-field
- vocabulary: closed | open (parametric)
- controls: <the ONE property/concept it sets>
- must-never-touch: <explicit out-of-scope list>
- default (unmarked): <value or "none">
- members: <the words, in order if a scale>
- notes / open rulings:
```

**Which bucket — default expectation.** An axis carrying a numeric ratio/count/degree
(`grow-N`, `span-N`, `exact-N`, durations) is **open (parametric)**; a perceptual or
platform-mirrored axis (a fixed scale, an ARIA state set, a named z-ladder) is **closed**.
This is the m2 ruling generalized ("ratio/weight is parametric, not perceptual"). The rule now
holds **without exception** across the grammar axes: both former against-the-grain candidates were
resolved by *removing* an axis rather than stamping one — *text/type* moved to skin (§5; not a
layout relationship), and *proportion* was retired (its anisotropy is per-side density composition).
A future axis should follow the default (numeric→open, perceptual/platform-mirrored→closed); a
genuine exception would itself be a `[RULING]`.

## 4. The grammar plane — four sibling grammars (settled)

The grammar plane comprises four siblings sharing the constitution (§§1–3):
**4.1 layout** (in-flow arrangement) · **4.2 state** (interaction) · **4.3 motion** (time) ·
**4.4 layering** (out-of-flow depth). Skin (§5) and identity (§8) are the other planes.

### 4.1 — LAYOUT grammar (in-flow spatial arrangement)
Axes pre-filled from current code; rulings flagged. Held distinct from 4.4 layering (out-of-flow).

### structure — container-role
- signature: container-operation
- vocabulary: closed (`horizontal/vertical/grid` exhaust the inner-display choices)
- controls: the **inner display type** (CSS Display L3) — how children arrange: flow / flex / grid (+ direction)
- must-never-touch: spacing, alignment, the element's own size/flex, appearance, the **outer**
  display type (block/inline — that's member-role, below)
- default: **flow** (normal flow) — RULED. `block`/`inline` is the *outer* display type (how the
  element sits in its parent) = **member-role**, NOT structure. Structure picks the inner type;
  the unmarked inner type is flow, with flex/grid opt-in (don't default to flex). NB: CSS's
  outer/inner display split *is* the member/container role split.
- members: `horizontal` `vertical` `grid`
- **retired: `rows`** (compose-don't-coin, §8). It expanded to `flex-direction: row` + `flex-wrap: wrap`
  — i.e. exactly `horizontal wrap-allowed`. It was a coined composition, and it wrote `flex-wrap`,
  which belongs to the **wrapping** axis, not to structure (whose mandate is inner-display-type +
  direction). The overlap was a latent dimensional-purity violation, hidden only because the wrapping
  axis had no emitted CSS yet; giving wrapping its emission surfaced it. Resolution: drop the word, keep
  the composition.
- frozen compositions (dissolve into grammar — see §8): `two-sections` and `sidebar-main`
  bake in structure + per-child member-roles; `sidebar-main`'s `:first-child`/`:last-child`
  rules are heterogeneous member-roles hardcoded.

### member-role — a FAMILY of sub-axes  *(was "sizing" + blocks/items/sections)*
*(vocabulary is stamped **per sub-axis** below — the family is mixed: m1/m4 closed; **m2** open +
whole-axis aliases; **m3/m5** closed + parametric member (§2); m2/m3 are the **negotiated** regime,
m1/m4/m5 are free.)*

**Essence (what the sub-axes share):** every member-role property is resolved *relative to the
parent's allocated space* and is **inert without a parent** — it's the element's *response* to
the slot its parent gives it, not a property of the element alone. Duality: container-role =
*supply* (the layout operation); member-role = *demand* (the operand's parameters to it).

**Discriminator test:** remove the parent (or give it a plain-flow parent) — does the property
still mean anything? **No → member-role.** Sets up something for *children* → container-role.
Neither → self/internal (padding) or another plane.

Enumerated, this is **not one axis but ~5 orthogonal sub-axes** — they compose on one element
without conflict (unification passes). Shared `must-never-touch`: children, arrangement, spacing,
appearance.

#### m1 — flow participation (outer display) · `set-with-exclusivity` · **vocabulary: closed** · default: natural (override-only)
`inline` (display:inline) · `boxed` (block) · `boxed-inline` (inline-block). Leading candidate
(RULING open). `boxed` over noun `box` (which reads as the ecosystem `<Box>` + box-model echo);
box-ness isolates `inline`, the `-inline` suffix carries the flow distinction.
*(Surface-name candidate flagged for audit, NOT changed now: `boxed` may still read as visual —
bordered/card-like — to some authors. Alternatives that are clearer about mechanism but worse as words:
`block-flow` / `flow-block` / `as-block` / `outer-block` / `block-member`. Per the alias law, settle
this from observed recurrence in real components, not from argument — it is a guide-surface name, not a
structural choice.)*

> **Outcome constraint — m1 is inert on a flex/grid item (browser-verified, `demo/` + `test/browser`).**
> CSS *blockifies* the outer display of a flex/grid item, so `inline` / `boxed-inline` are **no-ops on a
> child of a flex/grid container** (`horizontal inline` in flow → `inline-flex`; the same as a flex item →
> `flex`). This is correct CSS, not a defect — but it is an *outcome* interaction P7 cannot see (no property
> collides; the parent's layout mode silently overrides the child's outer display). Enforced by **lint warning
> P11** (`flow-participation-inert`, `lint.ts`): it reads the parent's classes from `LintContext.parentClasses`
> and warns when an inline-flavored m1 word rides a flex/grid item. This is the first entry in the "outcome
> behavior" frontier the status note names — surfaced by rendering, not by the paper checks — and the first
> lint rule that consumes **parent context** rather than the class string in isolation.

#### m2 — flex character (give↔grab) · **2D product** (grow × shrink) · **vocabulary: open + whole-axis aliases** — open dials `grow-N`/`shrink-N`; 4 corners are whole-axis aliases (each fixes both dials) · **regime: negotiated** · default: compressible
Grow-willingness and shrink-willingness are *independent* → a product, NOT a chain. Corners,
**all adjectives**: `rigid` (0,0) · `compressible` (0,1, shrinks) · `expandable` (1,0, grows) ·
`elastic` (1,1, both). The two atoms are `compressible` + `expandable`; `elastic`/`rigid` are
aliases for both/neither. Replaces old `block`/`item` and the register-inconsistent
`yielding`/`grow-only`. (Default = compressible = CSS `0 1 auto`; `rigid` needs the word.)
**Correction (playground-verified):** `flex-basis` is NOT part of give↔grab — `greedy` and
`elastic` differ *only* by basis (grow/shrink both 1,1), so on this axis they are the same point.
So **retire `greedy`** — it named a surplus *symptom*. `flex:1 1 0` = `elastic` (m2) + a
`ratio` basis (m3). And do NOT rename it for the deficit symptom (`collapsing`/`compacting`)
either: the deficit collapse-to-0 has **two distinct causes** (`ratio` basis vs `compressible`+`min:0` — see
m3), so a symptom-name would merge them. `basis` factors to m3 (the "normal" size). Surplus with one
grower *hides* the basis difference (smaller basis → larger free space, cancels); deficit *exposes*
it (shrink is weighted by basis → basis 0 has zero shrink-weight).
**Ratio/weight is parametric, not perceptual** (Jorge's "grow 1 vs 2"): the 4 corners are
*qualitative* (does it grow/shrink at all); *how much* relative to siblings is the **value on a dial**
— like grid's `fr` number or m5's `span-N`. Carry it on the numbered dial directly: `grow-2`.
> **RULED (whole-axis aliases — this revision; supersedes "corner + optional weight").** The earlier
> framing said `elastic` + `grow-2` — a corner *plus* a weight modifier — was the sanctioned form. That
> was wrong, and it was the source of a long detour. The justification is ordered below by what
> actually *forces* the conclusion (the early "a class string is a set, so order can't matter" argument
> is **not** load-bearing and is dropped: a `calc(var(--on,0)*var(--mag,1))` encoding is order-free,
> so order alone never settled it — the design choice did).
>
> 1. **The single invariant (decisive): no CSS property is written by more than one co-present class.**
>    This is just Law 3 / predicate-4 (property-disjointness) applied at *longhand* granularity inside
>    the negotiated axis. `flex-grow` and `flex-shrink` are **independent longhands**, so the dials are
>    independent for free: `grow-2` writes `flex-grow` only, `shrink-1` writes `flex-shrink` only —
>    `grow-2 shrink-1` composes natively, no machinery. A **whole-axis alias writes *both* longhands**
>    (`expandable` = `flex-grow:1; flex-shrink:0`). That breadth is exactly why an alias cannot sit
>    beside a dial: `expandable grow-2` writes `flex-grow` twice → the forbidden double-write. So
>    `elastic grow-2`, `expandable shrink-2`, `rigid grow-2` are all conflicts — and the whole-axis-alias
>    rule is a *consequence* of the one invariant, not a separate rule. (Emit **longhands**, never the
>    `flex` shorthand — the shorthand bundles all three sub-properties and would stomp `flex-basis`/the
>    other dial; that cross-property reset is a manufactured collision, avoided by using longhands.)
> 2. **The tell that the old split was wrong (historical, but the cleanest):** the prior model split m2
>    into two records both controlling `flex-grow`/`flex-shrink`, and needed a hand-written cross-class
>    predicate (P5) to keep `rigid` agreeing with the weights. *Genuinely independent axes never need a
>    bespoke rule relating them* — Law 3 gives that for free. The existence of P5 was the evidence the
>    two "axes" were one; collapsing them retires the predicate.
> 3. **Law 6 (why no gate variable either):** a "grows: yes/no" gate is **not a distinction CSS draws**
>    — `flex-grow:0` already *is* "doesn't grow." A two-facet (gate × magnitude) model mints that gate
>    as coined ontology and pays a `calc` for it, buying nothing the dials don't already give. So the
>    corners are earned *aliases* over the dials, not a separate "gate" layer.
>
> The rule, then: write **either** an alias (the common case) **or** the numbered longhand dials
> (`grow-2 shrink-1`) when a non-default magnitude is needed — never both. m2 stays honestly `open` (the
> dials are the substrate) with the corners as earned sugar (§2: open + whole-axis aliases).
>
> **Per-dial default — RULED (the `grow-2` surprise).** An unspecified dial keeps **its own CSS initial
> value** (the m2 default `compressible` = `flex-grow:0`, `flex-shrink:1` is exactly those initials).
> Consequence authors must be told: **`grow-2` is *not* grow-only** — it sets `flex-grow:2` and leaves
> `flex-shrink` at its initial `1`, so the item grows *and* shrinks (`grow-2` ≡ behaviourally
> `grow-2 shrink-1`). Grow-only at weight 2 is written explicitly `grow-2 shrink-0`; grow-only at weight
> 1 is `expandable`. (The default lives in CSS's initial values, one layer down — the grammar states it
> so the surprise is documented, not patched by teaching a single dial to mean two things.)
**Weight — RULED (layered: primitives + corner aliases).** `flex-grow`/`flex-shrink` are
independent, so the weight is **two primitives**, not one:
- **primitives:** `grow-N` · `shrink-N` (two independent scales, N≥0) — *complete*; express any case
  including asymmetric edges (`flex:2 1` = `grow-2 shrink-1`).
- **aliases:** the 4 corners are aliases over the primitives — `rigid`=`grow-0 shrink-0` ·
  `compressible`=`0,1` · `expandable`=`1,0` · `elastic`=`1,1`.

Grow-only is `grow-N`; asymmetric is `grow-N shrink-M` — that covers the real cases, so no
single symmetric weight alias is minted.

> **Alias law (Jorge):** *aliases are detected from repeated real usage, not invented speculatively.*
> Codify a shorthand only after a primitive combination demonstrably recurs — never the other way around.

Primitives are complete, aliases are kind. Consistency rule (now a special case of one-word-per-axis):
a whole-axis alias is a complete value, so it cannot combine with a numbered dial — `rigid grow-2`,
`elastic grow-2`, `expandable shrink-2` all conflict → lint. (`grow-2 grow-3` also conflicts: two
values on one dial.)

#### m3 — self size · `set` · **vocabulary: closed + parametric member** — closed {`content`, `ratio`, `exact`}; `exact` carries a size value · **regime: negotiated** · default: auto
Member-role self-size = `basis` (the **normal**/start size; flex-only; factored out of m2). The
*source* of `basis` is content/ratio/exact (below).

> **CORRECTION (predicate-4, collision 2 — this revision): m3 controls `flex-basis` ONLY; it does
> NOT control `align-self`.** The prior wording listed "`fill`/`stretch` (fill the parent)" as an m3
> size source. The static dimensional-purity check (§10.1) found this made `align-self` claimed by
> *both* m3 and m4 — a latent plane-mix. The "fill" wording conflated two different things, neither a
> distinct m3 responsibility: **main-axis fill** is just m2 `grow` (already covered), and **cross-axis
> stretch** IS m4's `align-self: stretch` (already covered). So m3 = `basis` (+ source) only; the
> "fill the parent" capability is recovered *unchanged* by composing m2 (`grow`, main-axis) and m4
> (`self-stretch`, cross-axis) — the textbook Law-6 case "the felt gap is a composition across
> existing axes." Verified: m3 ∩ m4 = ∅ after the correction.

**Edge — RULED (sharpened by the §6.6 reconstruction).** Discriminator: *flex-only vs universal box
property.* `flex-grow`/`shrink`/`basis` are inert without a flex/grid parent → **member-role**
(`align-self` is m4's, not m3's — see correction above); `min-width`/`max-width`/`width` are
**universal** box properties (meaningful on any element; the flex algorithm only *reads* them as
result-clamps) → **self/internal**. Therefore:
- the **`min`/`max` bounds** (and `width`) → **self/internal**, whatever their value.
- `fit`/`min`/`max-content` are intrinsic *values* (self/internal measurement vocabulary), **not an
  axis** — usable as a self/internal bound (`min-width: min-content`) OR as a content-*source* for the
  member-role `basis` (`flex-basis: max-content` = a finer `content`).
- member-role keeps **give↔grab (m2) + `basis`(+source) (m3) + cross-axis `self-stretch` (m4)**.

**Cross-plane "band":** the min/normal/max model survives but spans *two* planes — the *band*
(`min`..`max`) is self/internal; the *normal* (`basis`) and the *force* (m2 give↔grab — shrink pulls
toward min, grow toward max) are member-role. The §6.6 pipeline fuses inputs from both planes into one
width: the canonical "orthogonal-to-author, entangled-in-computation" case.

**Basis source — RULED. Formal class surface is `basis-*` (the bare word is the *source value*, not a
standalone class).** `basis` answers *where my size comes from*, with **three sources**; the authored
class always carries the `basis-` prefix so the word is self-disambiguating (a naked `content` or
`ratio` is too broad to stand alone — it is only clear scoped by `basis`):
- **`basis-content`** (`basis:auto`) — size to my content (Figma *Hug*).
- **`basis-ratio`** (`basis:0`) — sized by my grow ratio = a share of the space (flexbox's `fr` /
  Figma *Fill*).
- **`basis-exact-N`** — a precise declared size; the `N` is parametric (like m2's weight).
  Named `exact`, **not `fixed`** — `fixed` would collide with `position:fixed` (layering).
  > **RULING (value domain — this revision, prompted by the spec extraction): `N` is a SIZE-SCALE
  > TOKEN, not a raw length.** The prior wording `basis:Npx` left the domain open, which would have
  > admitted `basis-exact-240`, `basis-exact-1.5rem`, `basis-exact-[var(--x)]` — the same "raw
  > integers" smell the **z-scale** (§4.4) and **density** (§4.1) both forbid. For consistency with the
  > grammar's "named scales, never raw values" stance, `basis-exact-*` references the **§5.1 size scale**
  > by step name: `basis-exact-sm`, `basis-exact-md`, `basis-exact-lg` (the generator-defined scale,
  > swappable per theme). **Arbitrary CSS lengths are OUT of the grammar surface in v0** — an exact
  > pixel/var size is an identity/one-off concern (it goes in component CSS, not a grammar class), the
  > same way `equal-square`'s fixed `3rem` was ruled an identity/tokenization matter, not a grammar
  > primitive. If a raw-length escape hatch is ever wanted, it returns through the alias law
  > (recurrence first) as an explicitly-marked arbitrary-value form, never as the default. So the
  > **axis is `closed`** (membership {`content`, `ratio`, `exact`} is exhaustive — see §2) and **the
  > `exact` member is parametric over a *closed* value space** (the size-scale tokens, not arbitrary
  > lengths). This is the "closed axis, one parametric member" shape of §2; the earlier note that read
  > "`vocabulary: open` still holds" was from before the vocabulary-shape settlement and is **withdrawn**
  > — m3 is closed, with a closed-parameter member. Linter pattern: `/^basis-exact-(sm|md|lg|…)$/`
  > against the live scale step names, not `\d+`.

The bare source words (`content` / `ratio` / `exact`) name the *value space* of the `basis` concept;
they are **never** emitted as standalone classes. Reads as "sized by **content** / by **ratio** / to an
**exact** value." Useful `ratio` case =
equal/proportional columns (`flex:1 1 0` ≈ `1fr`), content-independent — a *sizing* feature, **not**
a "collapse". The deficit collapse-to-0 has **two causes that must stay distinct**:
(a) `ratio` (basis 0, no surplus to grant a share — never had intrinsic size); (b) `compressible`
+ `min:0` (had a size, gave it away). Same symptom, opposite meaning → name the cause, not the symptom.

#### m4 — self-alignment (cross-axis override) · `set-with-exclusivity` · **vocabulary: closed** (mirrors `align-self`) · default: inherit group
`self-start` · `self-center` · `self-end` · `self-stretch` · `self-baseline` (align-self). The
**member twin** of the container `align-*` axis — same words, opposite role.

#### m5 — grid placement · positional (not perceptual) · **vocabulary: closed + parametric member** — closed {`span`, `row-span`, `span-all`}; `span`/`row-span` carry an integer, `span-all` contextual · default: auto-place
`span-all` (grid-column:1/-1) · `span-2` · `row-span-2`. Only meaningful under a `grid` parent.
NB: member-role mixes *perceptual* sub-axes (m2/m4) with *positional* ones (m5) — don't force
perceptual words onto numeric placement.

**Cross-cutting rulings (carried over):**
- **dedupe — RULED:** the old bundled members minted twins — `item`/`fit-content` were both
  `0 0 auto` (= the unified `rigid`); old `fill`/`greedy` both grabbed space (now `expandable`/
  `elastic`+`ratio`). The m1–m5 split + new names resolve them. Live v0: `.items` was an unused
  duplicate of `.fit-content > *` → removed.
- **declaration site — RULED:** *default + override*. The container supplies the normalized
  member-default (distributed); the element asserts its singularity locally (Law 4 across the
  tree). Mechanism: zero-specificity distribution `:where(.blocks > *){…}` + `.item{…}` so the
  override always wins. **Depth RULED:** container scope = one level (`> *`); wider "across-all"
  = substrate/ambient scopes (§1), not container grammar.
- **`sections` — RULED: dropped.** Empty, unused vestige; "major divisions" belong to HTML
  semantics (`<section>` / landmarks), or to density (`separated`/`loose`) + structure, or a
  composition — not a grammar primitive. (`.two-sections`, a real grid layout, is unrelated and stays.)
- **`equal-circle` — RULED:** it was `equal-square` (size) **plus** `border-radius:50%` (skin) — a
  plane-mix → by the decomposition test, a circular shape = size grammar + a skin (`circle`/`rounded`).
  Live v0: `.equal-circle` was unused → removed. `equal-square` stays (pure size; its fixed `3rem` is
  an identity/tokenization matter, not a plane-mix).

### space ownership — gap / flow / padding / margin (RULED)
*(not a vocabulary axis — this is a **role assignment**: which property owns which spatial
relationship. The spacing *scales* and their open/closed status live in density / padding above.)*
The three "space" properties don't overlap once each owns the *relationship* it describes:
- **gap** = space BETWEEN siblings → **container-role** (the `density` axis below). Default
  owner of inter-element rhythm; no collapsing, no first/last-child stripping.
- **padding** = space WITHIN self (and a container's inset from its own edge) → internal/self.
- **margin** = space between self and the OUTSIDE → **member-role, outward**. *Margin-between-
  siblings* is a member doing the container's job (leaked concern, and it collapses) → use
  `gap`/`flow`. *Margin-on-self* is the legitimate member-role use (below).

**Rule — RULED (L271, revised L304).** Four spacing families, each a **per-property class** over the
one shared scale (§6); the property + where it sits picks the role:
- **`gap-*`** (container) — inter-sibling rhythm in flex/grid; the default for inter-element space.
- **`flow-*`** (container) — inter-sibling rhythm in **block/prose flow** where `gap` can't reach
  (the owl `> * + *`, ≈ Every Layout's *Stack*). Replaces ad-hoc prose margins:
  `.content-editor-body p { margin-* }` → `flow-comfortable`.
- **`padding-*`** (self) — internal space / a container's own inset.
- **`margin-* / margin-inline-* / margin-block-*`** (member, on the element itself) — an element's
  *own* outward space (a bordered chip; a callout). **First-class and directional**, but **marked by
  preference**: reach for it only when an element needs space independent of any container rhythm.
  Also the home of `margin:auto` distribution and negative/overlap space (which `gap`/`flow` can't do).

All four are **independent** on one element (`padding-comfortable margin-snug` just works) and stay
**single-sourced** (they read the same scale). The full property name + density word is the
convention (`margin-inline-snug`, not `snug-margin`).

> **`flow` vs `stack` — RESOLVED by dissolution (no new word).** There is **no separate structure
> member** to name: vertical stacking of children is the existing **`vertical`** member (partner to
> `horizontal`). `flow` is reserved for the **unmarked structure default** only (CSS `display: flow`,
> mirroring the platform — Law 6), so it never doubles as a marked member (which would violate
> markedness, Law 4). `stack` is not needed and is not minted. NB: this is unrelated to the
> **`flow-*` spacing axis** below (inter-sibling rhythm in block/prose flow) — that keeps its name;
> the collision was only ever between *default* `flow` and a hypothetical *member* `flow`, now removed.
> Markedness reads cleanly: bare = `flow` (default) · column = `vertical` · row = `horizontal`.

### density (behavior) — gap
- signature: ordered-chain (scale)
- vocabulary: closed (the fixed scale; a new step is a `[RULING]`, never a coin)
- controls: `gap`
- must-never-touch: padding, margin, structure
- default: **`comfortable`** — RULED (provisional; revisit if a different neutral feels right).
- members (order): `tight` `snug` `comfortable` `relaxed` `loose` `separated`
- **open rulings (this axis has the known smells):**
  - **`relaxed`/`spaced` — RULED:** both were `--spacing-lg` → Law 5 synonym. **Collapse: keep
    `relaxed`** (the canonical density word), drop `spaced`. Live v0: `.spaced` was unused → removed.
  - **`comfortable-relaxed` — RULED: DERIVE it, don't hand-author it.** Same multiplicative-vs-
    additive + single-source argument: a static class per density-pair is **D²** with the values
    duplicated; instead the pair derives from atomic per-axis density (per-side per-property classes,
    `padding-inline-*`/`padding-block-*`, over the one scale) → **additive (2D), one source of
    truth**. So `comfortable-relaxed` is a
    *derived/compiled alias* (≈ `comfortable-y` + `relaxed-x`), never a primitive. **General rule
    (Jorge): what is safe to be derived should be derived** — compile composites from atoms rather
    than enumerate hand-authored ones.
  - **Unify density + padding — RULED: YES**, one shared **scale** referenced by per-property classes
    (§6): `--d-comfortable` defined once; `padding-comfortable`/`gap-comfortable`/`margin-comfortable`
    each read it (**single source**, full independence). The runtime value-channel that earlier did
    this is **RETIRED** (see §6.3, L304); `valueChannel.browser.test.ts` now tests a retired
    mechanism → repurpose or remove.

### padding (behavior) — RULED: per-property `padding-*` class over the shared scale
- signature: ordered-chain (scale) — *the same density words, applied to `padding`*
- vocabulary: closed (shares density's fixed scale)
- controls: `padding`
- expression: **`padding-<density>`** reading the one scale (§6) → `padding-comfortable`,
  `padding-snug`, … (full property name + density). Independent of `margin-*`/`gap-*` on the same
  element; single-sourced via the scale var.
- **`wide` — RULED:** a *proportion* modifier (horizontal bias), **not** a padding step. With the
  channel retired, per-axis bias is expressed via per-side classes (`padding-inline-*` /
  `padding-block-*`); see proportion axis (its mechanism is now reopened).

### proportion (behavior) — **RETIRED as an axis** (was: the Le Corbusier thread)
**Resolution (R3):** proportion is **dropped as a grammar axis.** The capability it named —
anisotropy, a different ratio on each spacing/sizing axis — survives in full as **per-side
per-property density classes** (`padding-inline-relaxed` + `padding-block-snug`, etc.) over the
one scale (§6). So nothing is lost but the single-word reading; the distinction proportion drew is
*already expressed* by composing two existing density members, which is exactly the "felt gap is a
composition across existing axes" case of **Law 6**.

Why retired, on the record:
- its **mechanism** was the runtime value-channel (`--space-x: calc(var(--space) * ratio)`); when
  the channel was **retired** (§6.3), proportion lost its only implementation, and per-side classes
  already cover the need — a capability with no unique residue is not an axis (cf. §6.3's reasoning).
- `square`/`wide`/`tall` were therefore only ever **sugar over per-side density**; by the alias law
  they would have to be *earned* by recurrence, and they never were — so they are not minted.
- **`derive what is safe to derive` (Jorge):** `wide` ≡ "more inline than block padding" is a
  derived alias, not a primitive; the grammar prefers the per-side atoms.

*Design history kept for reuse:* the **rejected 6-word perceptual chains**
(`open/spread/wider/broad/extended`, `upright/raised/elongated/taller/lofty`) failed for reasons
worth remembering against any future bipolar axis — register-mixed, non-monotonic/synonymous,
several named the wrong concept, and `raised` collided with the layering z-scale. If a single-word
proportion reading is ever genuinely wanted again, it returns through the **alias law** (recurrence
first), as sugar over per-side density — never as a re-minted primitive axis.

### alignment (behavior)
- signature: set-with-exclusivity (two sub-axes: main / cross)
- vocabulary: closed (the fixed `align-*` / `justify-*` keyword sets)
- members: `align-start|center|end|stretch|baseline` · `justify-start|center|end|between|around`
- must-never-touch: spacing, structure

### divider (behavior) — container-role  *(RULED: the stroke-twin of `gap`)*
- signature: set-with-exclusivity
- vocabulary: closed (`divided` / `undivided`)
- regime: free
- controls: a **separation line drawn *between* sibling members** — present between adjacent
  members, absent before the first and after the last.
- **Ruling (Jorge):** *the element alone must not carry a line below it; it is the **group**
  (the container) that requires the separation mark.* So divider is **container-role**, never
  member-role — the exact `gap` logic applied to stroke instead of space: a between-siblings
  relationship the container owns and distributes, not a property any member declares.
- must-never-touch: the members' own borders/box (a member's *own* outline is skin), spacing
  (that's `gap`/density), structure, appearance beyond the single separating stroke.
- default (unmarked): `undivided`.
- members: `divided` (lines between members) · `undivided` (none — default).
- **Law 6 basis:** this is a distinction the platform draws — the **CSS Gaps Module L1** "gap
  decorations" (`column-rule` / `row-rule` / `rule` shorthand: *a line painted in the gap, between
  adjacent siblings but not at the container edges*) — given a perceptual name; not invented
  ontology. The spec's own framing ("spacing between adjacent sibling boxes, distinct from spacing
  at the first/last box and the container edge") **is** this ruling stated in CSS.
- **anti-pattern it replaces:** today the sibling-divider is faked with `border-bottom` on every
  member (then fighting the last-child border) — the stroke version of faking `gap` with
  per-child `margin-bottom`. `divided` pulls the relationship up to the container where it belongs;
  no per-child borders, no `:last-child` exception.
- **mechanism — open (Tier-2, non-blocking), with a CURRENT default that is CONDITIONALLY correct:**
  the forward path is the native gap-decoration properties (`row-rule` for vertical-stack lists,
  `column-rule` for rows) in flex/grid containers — but those are **Chromium-only as of mid-2026**
  (Chrome/Edge 149; Firefox and Safari not yet), so they are **not baseline**. Note: `column-rule`
  *in multi-column layout* is long-baseline, but `column-rule`/`row-rule` *between flex/grid items* is
  the new, not-yet-portable part — and the latter is what `divided` needs. **Current default
  mechanism:** a generated `:where(.divided > * + *)` leading-border.
  > **CORRECTION (weakness-#5, this revision): the `* + *` fallback is correct ONLY for single-line,
  > source-ordered containers** (no `wrap-allowed`, no `order`, no reversed direction). `* + *` paints
  > on **DOM-adjacency**; under wrapping it strokes the first item of every *visual* row (and omits the
  > stroke *between* rows), and under `order`/reverse, DOM-adjacency diverges from visual-adjacency so
  > the stroke lands between the wrong pairs. The native `row-rule`/`column-rule` path is correct
  > precisely because it paints in the **resolved gap** (post-wrap, post-order). This is itself a small
  > Law-6 lesson: `* + *` is a CSS idiom naming *DOM-sibling adjacency*, which is **not** the
  > relationship `divided` means (*visual* between-ness); gap-decorations name the right line.
  > **Therefore:** `divided` + `wrap-allowed` (or any `order`/reversed member) **requires**
  > gap-decoration support; where unavailable it must **degrade to no divider** (safe) rather than
  > mis-render (wrong). The linter warns on this composition (§9.1). `migrating to` native
  > `row-rule`/`column-rule` once cross-browser. This is a *value/implementation* ruling (like the
  > motion token scale), not a *placement* one; the role/signature/vocabulary above are **frozen**.
  > `<hr>` is unrelated — that is an **element** (thematic-break content), styled by skin, not this
  > relationship.
- **stroke skin is late-bound:** `divided` owns *that* a line exists between members; its color /
  thickness / style are **skin tokens** (like any border's look), not part of this grammar axis.

### wrapping (behavior)
- vocabulary: closed (`wrap-allowed/prevent/reverse` exhaustive)
- members: `wrap-allowed` `wrap-prevent` `wrap-reverse`

### overflow (behavior)
- vocabulary: closed (`scroll-y/x/auto`, `clip` exhaustive)
- members: `scroll-y` `scroll-x` `scroll-auto` `clip`

### constraints
- signature: set-with-exclusivity
- vocabulary: open (parametric) — `min-width-*` `max-width-*` `min-height-*` `max-height-*` each
  carry a value from the §5.1 size scale
- **four independent sub-dials**, one per longhand: `min-width` `max-width` `min-height`
  `max-height`. A min and a max on the *same dimension* form a **band** and compose
  (`min-width-sm max-width-lg`); two values on the same dial still conflict. All four dials may
  co-occur. No whole-axis alias — there's no single word that sets all four at once. (Same shape as
  the padding/overflow sub-dial generalization above — added this revision to fix a real gap the
  external review surfaced: the axis previously modeled min/max as plain alternatives, which
  wrongly rejected the ordinary case of bounding both ends of a dimension.)
- must-never-touch: everything except min/max bounds
- open item: a semantic check for an inverted band (e.g. `min-width-lg max-width-sm`) is not yet
  implemented — see spec §2.1.

### interaction → promoted to **§4.2 STATE** (feedback animations → **§4.3 MOTION**)

### text / type — **NOT a grammar axis → SKIN** (RESOLVED, R-type)
**Resolution:** type is **not layout grammar**; it moves to **skin (§5)**. The grammar *references*
type tokens late-bound (as it does color), but owns no type axis.

Why, by the document's own tests:
- **discriminator test (§4.1) → "neither":** remove the parent — `font-size` still means something
  (text has a size with no parent; it's a property of the element's *own content flow*), so it is
  **not member-role**; it sets up nothing for children, so **not container-role**. "Neither →
  self/internal or another plane" — type is **self/internal surface**, i.e. skin.
- **the relationship is internal to the text** (the flow and the space it occupies), **independent
  of the container** — even though the container then *reacts* to the text's final measured size.
  That reaction is the §6.4 **intrinsic-sizing** pattern: content size is an **input the layout
  solver reads** (`basis: content`), not a layout relationship the grammar owns. Type is *upstream*
  of grammar, the way content is.
- **"has a scale" ≠ grammar:** spacing has a scale and *is* grammar because `gap` is a relationship
  **between siblings**; `font-size` is a property of **one element's own text**. The scale-ness is
  the **§5.1 generative-proportional** kind (lawful skin), not a layout relationship.

Where each piece lands in §5:
- **`font-size` / `line-height`** → **§5.1** (generative-proportional): rebuild `text-xs…text-3xl`
  and line-height as a true **modular scale** (one base × one ratio) — the same rebuild §5.1 already
  targets for spacing/radius. Proportional, lawful, author-set.
- **`font-family` / `font-weight` / `text-mono`** → **§5.3** (connotative/learned): typeface and
  weight are taste/brand choices, late-bound and swappable (a theme = a position in taste-space).

Net: the former `text-*` / `font-*` utilities are **skin tokens**, not grammar members. This closes
the last grammar/skin straddle and removes a provisional axis rather than stamping one.

### 4.2 — STATE grammar (interaction · condition) — REBUILT on the full platform surface

> **Rebuild note (supersedes the prior §4.2).** The prior state set was enumerated from the
> states `.macro-search-item` happened to need, then declared *closed*. Round-2 audits (dialog,
> data-row, form-field) each forced a state addition (`open`, `aria-sort`, `invalid`/`required`),
> which is the signature of a set closed *prematurely* — closed against one component's needs, not
> against the platform's actual state surface. Law 6 demands the stronger move: **if states are
> mirrored from the platform, the mirror is the *whole* platform state surface, enumerated once.**
> This section re-derives the closed set from the two authoritative, finite, enumerable sources —
> the **ARIA 1.2 states** (the *state* subset of states-and-properties; *properties* and
> *relationships* are excluded by rule, below) and the **CSS Selectors L4 UI / validity / input
> pseudo-classes** — and freezes on *that* union. New members enter only when those specs add a
> state (Law 6); never coined.

**vocabulary: closed** — this sibling is the literal, load-bearing case of **Law 6**: states are
*transcribed* from ARIA + CSS UI pseudo-classes, never invented. Names the **conditions** (when a
rule applies), not effects (what changes); effects stay plane-tagged. A **sibling, not a plane**,
because it answers *when* and bridges to JS/ARIA.

- signature: **set-with-exclusivity per state-group**; states compose *across* groups
  (`hover` + `selected` + `invalid` co-occur); the resting state of each group = its unmarked
  default (markedness).
- composition: **`state × plane → rule`** (a state conditions a skin / layout / motion / layering
  change). The state class carries the *condition only*.

---

#### State is a CONDITION ONLY — never a bundled effect (RULED, unchanged — Law 1 / decomposition test)
A state class names *when*; it must **not** carry skin or layout *effects*. The `selectable-group`
plane-mix splits into `selectable` (state) + `selection-subtle` (skin), composed via
`state × skin → rule`. (Unchanged from prior §4.2; see §9 audit.)

---

#### Three kinds of state — arity is a TRICHOTOMY, not a dichotomy (REVISED — R-arity, Audit C)
The prior §4.2 ruled states are "two kinds: discrete / continuous." Audit C (`aria-sort`,
`aria-checked[mixed]`) shows a third. The platform draws **three** arities, and the registry tags
each state with its arity so the linter knows its value-space:

- **binary** — on/off (`hover`, `selected`, `disabled`, `open`, `busy`, `invalid`, `required`).
  Backing is present/absent (or `="true"`/`="false"`).
- **enumerated** — a small *closed* value set, > 2 (`aria-sort` ∈ {none, ascending, descending};
  `aria-checked` ∈ {true, false, mixed}; `aria-current` ∈ {page, step, location, date, time, true,
  false}). A binary model cannot express `mixed` / `none` / `ascending`; this arity is required to
  hold them. **This is `set-with-exclusivity` at the value level** — one value per enumerated state,
  exactly the one-word-per-axis law applied inside a single state.
- **continuous** — a normalized 0→1 value (`scroll-progress`, `drag-progress`; and the ARIA range
  values `aria-valuenow`/`min`/`max` when surfaced). The motion-clock driver (§4.3).

All three are **inputs**; the difference is only arity. (R1's "discrete vs continuous" is preserved
and *refined* — "discrete" splits into binary + enumerated.)

---

#### Three driver kinds — what supplies the condition (REVISED — R-driver, Audit B)
Audit B (responsive) showed the grammar had **no home for breakpoints** — and that the fix is
*already prefigured* by §4.3's "driver-is-state" ruling, which dissolved scroll-driven motion into
`continuous-state × motion`. A media query is the same shape: a **condition that selects which rule
fires**. So state carries **three driver kinds**, distinguished by *what the condition reads*:

- **interaction-driven** — read from user action / widget truth: ARIA states + CSS UI pseudo-classes
  (`hover`, `selected`, `invalid`, `open`, …). The bulk of the set.
- **input-driven (continuous)** — read from a continuous input position (`scroll-progress`,
  `drag-progress`). Clocks motion (§4.3) instead of selecting a rule.
- **environmental-driven** — read from the *rendering environment*, not the element:
  - **viewport** (`@media (width …)`, orientation) — `viewport-≥md`, `viewport-portrait`
  - **container** (`@container`) — `container-≥md`; structurally *container-role* (conditions on the
    container's size, distributes to children — the one driver that is inherently container-role)
  - **preference** (`@media (prefers-*)`) — `prefers-reduced-motion`, `prefers-color-scheme-dark`,
    `prefers-contrast-more`, `prefers-reduced-transparency`

> **The environmental driver retroactively homes two orphans.** `prefers-reduced-motion` (prior
> §4.3 said "a condition → lives in state" but named no category) and `prefers-color-scheme` were
> floating. They are environmental-driven states. That a freshly-named category *absorbs existing
> orphans* is the evidence it is the right cut, not an ad-hoc patch. **A responsive layout is
> `viewport-≥md × horizontal` (R-driver) — no fifth sibling.** State was already the condition layer
> for layout, motion, and now environment; that load is exactly why it is the most structurally
> central sibling, and why it needed this rebuild.

---

#### Mirror boundary — what is a STATE vs what is OUT (RULED — the inclusion test)
ARIA's "states and properties" table is **not** all states. The grammar mirrors only the **state**
subset and explicitly excludes three neighbouring kinds the platform lists alongside states:

1. **Relationships / wiring are OUT** (R-wiring, Audits A + D — earned by recurrence).
   `aria-controls`, `aria-describedby`, `aria-labelledby`, `aria-owns`, `aria-activedescendant`,
   `aria-flowto`, `for`, and focus-management / `inert` are **imperative semantic wiring**, not
   conditions. They assert *which element relates to which*, never *when a rule applies*. This is
   the **state-sibling twin of §4.4's declarative/JS boundary**: an automated consumer must route
   these to authoring/JS, never to a state member. (Two audits hit this class; per the alias law's
   inverse, an *exclusion* earns a clause once it recurs.)
2. **Quantitative properties are OUT as binary states, IN only as continuous/parametric** —
   `aria-valuenow/min/max`, `aria-posinset`, `aria-setsize`, `aria-colindex`, `aria-level`,
   `aria-rowcount` are numbers, not conditions. A range value surfaces as a *continuous* state
   (driver clause) if motion/skin needs it; positional counts are content, upstream of grammar
   (like text, §4.1 R-type). Never a binary state member.
3. **Live-region politeness is OUT** — `aria-live`, `aria-atomic`, `aria-relevant`, `aria-busy`'s
   announcement role: these instruct *assistive-tech announcement scheduling*, an output concern
   with no visual rule to condition. (`aria-busy` is kept **only** for its visual sense — "this
   region is updating" → a loading skin — under the availability group; its live-region duty is out.)

The inclusion test, stated once: **a platform attribute is a grammar state iff it names a
*condition under which a visual rule should differ* — answers "when," is read (not asserted as a
relationship), and has a perceptual consequence.** Everything in the ARIA table failing that test is
property/wiring/announcement, not state.

---

#### Law 6b — mirroring is rebuttable by Contrast (NEW — weakness #3)
A platform distinction is *eligible* (Law 6) but not *automatically admitted*. If two platform-named
states produce no difference the grammar can act on — same effects, same composition, same entailment
surface — they **fail Contrast (Law 5) and collapse to one grammar member carrying the richer
backing as an entailment *set***, not two members. Mirroring imports the platform's *distinctions*,
not its *redundancies*.
- **Guardrail:** Law 6b can only ever *reduce* member count, *never increase* it, and *never coins* —
  it picks the better of two existing platform names and folds the other into the entailment set.
  Strictly weaker than coining: a merge, not a mint.
- **Worked case — `pressed` ⊑ `selected`.** ARIA splits `aria-pressed` (toggle buttons) from
  `aria-selected` (set items). Both are, in this grammar, the same state-instance: an "on/chosen"
  condition that conditions a selection skin and entails platform backing. They produce the same
  effect surface → Contrast collapses them to **one** member (`selected`) whose entailment is the
  **disjunction**, role-picked: `selected ⟹ aria-selected ∨ aria-pressed ∨ :checked`.
- **Consequence for the schema:** `state-instance` carries a backing **set** (any-one satisfies),
  not a backing atom. The §4.2 entailment linter already supports this (below).

---

#### State entails its platform backing (RULED — strengthened with the set form)
A visual state class **requires** its ARIA/DOM/CSS truth, or it renders a state *visually true but
semantically false*. The linter enforces entailment; a state-instance with no backing is **ill-formed**.
**Four** registry categories keep the rule from firing on the wrong class (RULED; the 4th added by
Audit E — combobox):
- **state-capability** (`selectable`) — container declares its members *can* enter the state
  (distributed **down**). Asserts no instance truth → **entails nothing**.
- **state-instance** (`selected`, `expanded`, `open`, `disabled`, `busy`, `invalid`, `checked`,
  `sorted` …) — element asserts *its own* state. **Entails** its backing **set** on **itself**
  (weakness-#3 set form): satisfied by *any one*.
- **state-conditioned skin** (`selection-subtle`, `invalid-ring`) — defines *how* a state looks.
  Conditioned by instance truth, asserts nothing → **entails nothing**.
- **state-relational** (`active-descendant`) — container asserts that *this specific member* is in
  the state **right now** (asserted **down at one member**). **Inverted entailment:** the backing is
  on the **container**, not the element — `active-descendant` on an option ⟹ the container has
  `aria-activedescendant` pointing at *this option's id*. NEW category (Audit E).

> **Why `state-relational` is a real fourth category, not a patched instance (Audit E — combobox).**
> The arrow-key-highlighted combobox option is in a condition that is **not** `focus` (DOM focus
> stays on the input), **not** `selected` (unchosen — `aria-selected` would be semantically false),
> **not** `hover` (keyboard, not pointer), and **not** `current` (transient pre-selection, not
> persistent location). Its only backing is the container's `aria-activedescendant` pointing at it —
> the element itself carries no state attribute. The `instance` category *cannot* express this
> (self-backing is never present on the option, by construction). It is the **mirror-twin of
> `selectable`**: capability is the container distributing *potential* down to all members; relational
> is the container asserting an *instance* down at one member. The rebuild added the capability twin
> and missed the instance twin; this closes the asymmetry. One member today (`active-descendant`),
> but `aria-activedescendant` roving recurs across **all composite widgets** (listbox, grid, tree,
> menu, combobox), so it is not a combobox one-off.
>
> **Linter consequence:** the entailment predicate is now *category-dispatched* — `instance` checks
> self-backing present on the element; `relational` checks `container[attr] === member.id`. Verified
> by the session script `combobox-audit.ts` (manifest; **not yet committed** to
> `src/styles/grammar/` — promotion tracked in §10.1).

---

#### Canonical state vocabulary — the full mirrored set (RESOLVED — R2-full, by Law 6 = transcription)
Grouped; each member tagged `(arity · driver · backing-set)`. **Backing-set** lists the platform
truths any-one of which satisfies entailment. Resting default per group is the unmarked case.

**focus group** *(interaction)* — resting = none
- `hover` (binary · `:hover`)
- `focus` / `focus-visible` (binary · `:focus` / `:focus-visible`)
- `active` / `pressed-momentary` (binary · `:active`) — the transient press, distinct from the
  *toggle* `pressed` which Law 6b folded into `selected`.

**selection group** *(interaction)* — resting = unselected
- `selectable` (capability · distributes condition)
- `selected` (binary · {`aria-selected` ∨ `aria-pressed` ∨ `:checked`}) — Law 6b merge
- `checked-mixed` (binary · `aria-checked=mixed` / `:indeterminate`) — the tri-state value
  `selected` can't hold; the one place the selection condition is enumerated, not binary
- `current` (enumerated · `aria-current` ∈ {page, step, location, date, time, true}) — "the one in
  the set you're at"; nav/stepper/calendar. Distinct from `selected` (Law 5: "where you are" ≠
  "what you chose").

**availability group** *(interaction)* — resting = enabled / idle / writable
- `disabled` (binary · {`:disabled` ∨ `aria-disabled`})
- `read-only` (binary · {`:read-only` ∨ `aria-readonly`}) — operable-but-not-editable; the platform
  draws this apart from `disabled` (read-only is still focusable/copyable). Law 6 ⇒ separate member.
- `busy` (binary · `aria-busy`) — *visual* "updating" sense only (loading skin); live-region duty out

**disclosure group** *(interaction)* — split by mechanism (Audit A)
- `expanded` (binary · `aria-expanded`) — **in-place** disclosure (accordion, combobox, tree).
  resting = collapsed.
- `open` (binary · {`[open]` ∨ `:open` ∨ `:popover-open`}) — **top-layer / details** presentation
  (dialog, popover, `<details>`). resting = closed. **The dialog state the prior set lacked**
  (Audit A). `expanded` ≠ `open`: ARIA + CSS both draw these as different lines (in-place grow vs
  top-layer present), so Law 6 ⇒ two members, not one.

**validity group** *(interaction)* — resting = valid / writable *(Audit D — the missing family)*
- `invalid` (binary · {`:invalid` ∨ `aria-invalid`}) — constraint-invalid (may be true on load)
- `user-invalid` (binary · `:user-invalid`) — invalid **after interaction**. The platform draws this
  apart from `invalid` precisely so error skin appears only post-interaction; Law 6 ⇒ separate member.
  *(Without this, `invalid`-on-load lights every empty required field red immediately — the exact UX
  the platform added `:user-invalid` to prevent.)*
- `required` (binary · {`:required` ∨ `aria-required`}) — resting = optional
- `out-of-range` (binary · {`:out-of-range` ∨ `aria-valuemin`/`max` breach)) — numeric-range breach;
  platform-distinct from generic `invalid`.

**sort group** *(interaction)* — resting = unsorted *(Audit C)*
- `sorted` (enumerated · `aria-sort` ∈ {none, ascending, descending}) — column sort indicator.
  Enumerated, not binary — the value carries the arrow direction. Surface spelling of the three
  values (`sorted-asc`/`sorted-desc`/`sorted-none` vs a parametric `sorted-<dir>`) is a Tier-2
  guide-name ruling, non-blocking.

**drag group** *(interaction)* — resting = at-rest
- `dragging` (binary · pointer-drag / `@drag`) — note ARIA's `aria-grabbed`/`aria-dropeffect` are
  **deprecated** (confirmed: removed in ARIA 1.1+), so the backing is the pointer/DnD platform event,
  not the dead ARIA attrs. Law 6 mirrors the *live* platform line only.

**continuous-input group** *(input-driven)* — resting = at-origin (0)
- `scroll-progress` (continuous · `scroll()` / `animation-timeline`)
- `drag-progress` (continuous · pointer 0→1)

**environmental group** *(environmental-driven)* — resting = base/unconditioned *(Audit B + orphans)*
- `viewport-<bp>` (binary-per-breakpoint · `@media (width)`) — the breakpoint set is a Tier-2 scale
  (mirrors §5.1's generator stance: breakpoints are a *named scale*, values open/measured, structure
  frozen). `viewport-portrait`/`-landscape` (binary · `@media (orientation)`).
- `container-<bp>` (binary-per-breakpoint · `@container`) — **container-role** (conditions on
  container size). The one environmental member that distributes like `selectable`.
- `prefers-reduced-motion` (binary · `@media`) — **homes the §4.3 orphan**
- `prefers-color-scheme-dark` (binary · `@media`) — homes the theming orphan
- `prefers-contrast-more` / `prefers-reduced-transparency` (binary · `@media`)

New members enter **only** when ARIA / CSS-UI / the media-query spec adds a state (Law 6); never coined.

> **RULING (environmental-state SYNTAX — this revision; structurally significant, flagged by the spec
> review). Environmental state is a CONDITION SCOPE expressed as a VARIANT PREFIX, not a bare class
> word — and this refines how Law 2 reads.** The review correctly observed that `viewport-md` is
> *conceptually* a state but *syntactically unlike* `selected`: it does not assert a condition true of
> *this element right now*; it names **which environment selects which rule fires**. Three of the four
> candidate surfaces (bare class token / selector-condition / variant-prefix / condition-object) were
> weighed; the resolution:
> - **Authored surface = variant prefix** (`viewport-md:vertical`, `container-lg:horizontal`,
>   `prefers-reduced-motion:no-motion`). The prefix is the **condition**; the word after the colon is
>   an ordinary grammar word from some axis. This mirrors the platform 1:1 (`@media`/`@container`/
>   `@media (prefers-*)` *wrap* rules — they don't sit *on* the element), so it is Law 6-faithful: the
>   prefix names the platform's actual at-rule, the suffix is the rule it guards.
>   ⚠ **`no-motion` here is illustrative of the SYNTAX FORM only — it is not a registered word.**
>   Motion currently has no member representing "reduced/off" (see §4.3: `motion-micro`'s easing
>   words and `motion-macro`'s choreography words all still animate). Flagged by external review; this
>   is a real capability gap, not something to coin silently — it needs its own ruling through this
>   document before a concrete example can be given. Tracked in `SESSION-RESUME.md`.
> - **Interaction/input states stay bare** (`hover`, `selected`, `scroll-progress`) — they *are*
>   element conditions and compose as before. Only the **environmental** driver takes the prefix form,
>   because only it is a rule-wrapper rather than an element-truth. (The `:hover`/`@media` split in CSS
>   itself is exactly this line.)
> - **Consequence for Law 2 (one-word-per-axis) — AMENDED, on the record.** Law 2 now reads "at most
>   one word per axis **per condition scope**." `horizontal viewport-md:vertical` is **well-formed**:
>   two `structure` words, but in *different* condition scopes (base vs `viewport-md`), so they do not
>   collide — they are the *same* axis evaluated under *different* environments, which is the whole
>   point of responsive layout. `horizontal vertical` (both base scope) remains a conflict.
>   `viewport-md:horizontal viewport-md:vertical` (same scope, same axis) is a conflict. The linter
>   therefore **buckets words by condition scope first, then applies one-word-per-axis within each
>   bucket.** This is the one place a condition partitions the class string, and it is confined to the
>   environmental driver by the rule above.
> - **Registry shape:** environmental conditions are **not** axis members; they are a closed set of
>   **scope prefixes** (`viewport-<bp>`, `container-<bp>`, `viewport-portrait|landscape`,
>   `prefers-*`) with their own validation (the `<bp>` set is the §5.1-style named scale). A parsed
>   word becomes `{ scope: "viewport-md" | "base", axis, member, value? }`; P1 runs per `scope`.
> - **`container-<bp>` keeps its container-role note:** it conditions on the container's size and
>   distributes to children, so as a scope it attaches at the container, but the syntactic treatment
>   (prefix + guarded word) is identical.
> This is a **value/surface ruling with one law-amendment** (Law 2's per-scope reading); the *category*
> work (environmental = the third driver, no fifth sibling) is unchanged and stays frozen.

---

#### container-distributed vs element-local (RESOLVED — reuses Scope §1, extended by Audit E)
Default is **element-local**. The container→member axis now has **both twins**:
- **capability, distributed** (`selectable`, `container-<bp>`) — container establishes a *potential*
  for all members.
- **relational, asserted** (`active-descendant`) — container asserts *one specific* member's current
  instance state (inverted entailment, above).
All other states are element-local.

### 4.3 — MOTION grammar (time)
**Vocabulary looks mixed at a glance, but isn't a third primitive (RESOLVED below; spec §2.3
corrected to match this session).** The *easing* curve is a **closed** named vocabulary (Law 6,
mirrored from animation tradition) — that's the axis's own `vocabulary: closed`. The *magnitudes*
(duration/delay/stagger) are **open** skin scales, but they're **external references the emitted CSS
reads**, not members of the motion axis itself — same shape as any other axis pointing at a §5.1
scale. So each motion axis is still cleanly `closed`; nothing here needs a `"mixed"` registry
primitive. Motion is the **arrow between two states/configs**, never a static property — defined on
*pairs* (states = nodes, transitions = morphisms). It sits between state and the planes: a state
change → plane properties change → motion governs *how, over time*.

**Structure — RESOLVED: one atom, one grouping operator (macro = composed micro).**
A macro transition is **not a second kind of motion**; it is a *group of micro edges coordinated as
one unit* — the container/member pattern lifted to time. So motion has exactly one primitive and one
operation over it:
- **micro edge = the atom (member-level)** — one property's arrow between two states (`transition`).
  Parameters: **`duration`** (magnitude → skin scale), **`easing`** (closed curve vocabulary, below),
  **`delay`** (magnitude → skin scale), **`direction`** (enter/leave symmetry — closed small set:
  `symmetric` / `asymmetric`, since a state's enter and leave curves often differ).
- **macro = a container-operation over micro edges (container-level)** — coordinates a *set* of
  member-edges as one scene. Its own temporal vocabulary, which only exists at the group level:
  **`stagger`** (the offset between successive members' starts — magnitude → skin scale) +
  **`choreography`** (closed set: `together` / `sequence` / `cascade`). Mechanism is View Transitions
  API for scene-level, plain `transition` for single edges — but that is *implementation* of
  "single edge vs coordinated group," **not** a semantic split. The missing-feeling "thin" motion
  vocabulary was exactly this container tier (stagger/choreography) never being named.
  > **CORRECTION (predicate-4, collision 1 — this revision): `stagger` MUST NOT emit raw
  > `transition-delay`.** The static check (§10.1) found macro `stagger` and micro `delay` both
  > reaching for `transition-delay` — and unlike the `display` container/member twin (different
  > facets CSS resolves independently), these are the **same facet**: a parent distributing
  > `transition-delay` to stagger a child and the child setting its own micro delay **overwrite** (the
  > child wins, the stagger is silently lost). So `stagger` emits a **separate custom property**
  > (`--stagger`), composed *additively* into the child's delay:
  > `transition-delay: calc(var(--own-delay, 0ms) + var(--stagger, 0ms))`. Now member-delay and
  > container-stagger **add** instead of replace (verified: 200ms own + 3×60ms stagger = 380ms, both
  > preserved). This is the temporal analogue of how `gap` distributes without colliding with a
  > member's own `margin`: the container offset and the member value live in *different* properties
  > and compose by addition.

**Easing — RESOLVED: closed, by Law 6 (mirror the animation tradition, mint no new curves).**
Easing is **not** raw taste — it is an established language for bézier shapes with a decades-deep
tradition (ease-in/out, Penner, Material standard/decelerate/accelerate/emphasized, "slow in and slow
out"). The discipline already draws these distinctions, so the grammar *transcribes* them (as it did
ARIA states), it does not invent: **`decelerate`** (ease-out — things arriving/settling, the
workhorse) · **`accelerate`** (ease-in — things leaving) · **`standard`** (ease-in-out — both) ·
**`emphasized`** (the signature/spring curve). Closed vocabulary; new curves enter only if the
discipline names one. **Easing is clock-agnostic** — it is the shape of the function over its input
axis *whether that axis is time or a continuous input* (next clause), which is why it is the deeper,
mirrored vocabulary while duration is a mere magnitude.

**Driver — RESOLVED: the input source is a STATE concern, not a motion parameter.**
What *drives* an edge is not a property of the motion; it is **which state drives it**, composed via
the existing `state × motion → rule`:
- a **discrete** state (hover/selected/…) → **time-driven** motion (the default clock: the edge plays
  over its `duration` after the state flips).
- a **continuous** state (`scroll-progress` / `drag-progress`, §4.2 continuous-input group) →
  **input-driven** motion (the edge's progress is bound to the input's 0→1 value instead of to a
  duration). Same arrow, same easing curve, **different clock.**
So **scroll-/drag-driven animation needs no new motion vocabulary** — it is `continuous-state × motion
→ rule`. The driver lives in **state**; motion stays purely about the **curve**. (This is the temporal
echo of the `place ∘ negotiate` seam: the grammar names the curve declaratively; *what clocks it* is
supplied upstream, by state.)

- **magnitudes → skin scales (open):** `duration`, `delay`, `stagger` are generated token scales
  (`--motion-*`), almost certainly **modular** (one base × one ratio — see §5.1), the brand's motion
  signature; per-edge overrides allowed. Parallels `--space-base`. These are the only *open* part of
  motion, and they live in skin, not grammar.
- **`prefers-reduced-motion` — RESOLVED (unchanged):** a *condition* → lives in **§4.2 state**,
  conditions a motion change via `state × motion → rule`. Motion owns the edge; state owns the when.

**Frozen vs open.** Frozen: the one-atom/one-operator structure, the closed easing set, the
closed `choreography`/`direction` sets, and the driver-is-state rule. Open (Tier-2, non-blocking,
→ skin §5.1): the **numeric values** of the duration/delay/stagger modular scales and the final
spellings of the `--motion-*` tokens. No open item affects class-string *validity*.

### 4.4 — LAYERING grammar (out-of-flow depth)
**vocabulary: closed** — the named z-scale is closed by design ("never raw integers") and governs
**only the tier-2 in-page middle** (see resolution below); the "above-everything" names migrate to
**top-layer** mechanisms, and the host-beating constant is a single boundary value, not a scale member.
The second coordinate system the document model lacks. **`position` is the mode-switch:**
static/relative = in-flow (→ 4.1 layout); absolute/fixed/sticky = out-of-flow (→ here, layer space).
- **z as a NAMED ordered scale, never raw integers** (ordered-chain): `base · content · raised ·
  dropdown · sticky · tooltip`. Baseline = `base`, top = last; "who's on top" becomes a *legislated
  order*. This scale is **tier-2 only** and emits `z-index` within an `isolation: isolate` context.
  > **CORRECTION (this revision — internal contradiction the spec extraction surfaced): tier-1 names
  > REMOVED from the z-scale member list.** The prior list ended `… overlay · modal · popover ·
  > toast · tooltip`, which **contradicted this section's own Tier-1 resolution** (below): `overlay`/
  > `modal`/`popover`/`toast` are **top-layer mechanisms**, joined by *opening* (native
  > `<dialog>.showModal()`, the Popover API), and emit **no `z-index` at all**. Listing them as z-scale
  > members preserved exactly the "pick a bigger number" model Tier-1 was written to dissolve — an
  > author could read `modal` as a z rung. They are now homed in the separate **top-layer-mechanism**
  > set (Tier-1, below), which is not an ordered z-scale and emits no z. `tooltip` is the **one
  > judgment call kept in the scale**: a tooltip is frequently *not* promotable to the top layer (it is
  > often anchored content inside a component's own context), so it stays as the scale's ceiling; if a
  > given tooltip *is* promoted, it uses the top-layer mechanism like any tier-1 surface. Net: the
  > z-scale is the **tier-2 middle only**, and the three tiers no longer overlap in vocabulary.
- **stacking context = a local scope for z** (reuses Scope): `isolation: isolate` at component
  roots so each subtree's z-order is self-contained and the scale stays reliable.
- **quarantine the external "beat the host page" hack:** content-script overlays legitimately must
  outrank unknown host z — a SINGLE boundary constant at the shadow-root edge, NOT part of the
  internal scale. (Today: `z-index: 2147483646`.)
- members today (from §9 audit): `anchor` (position:relative context) · `isolate` · `layer-<level>`.
- disambiguation: CSS `@layer` = *cascade* layers (specificity), **NOT** z-index — different concept.
**z-mechanics — RECONSTRUCTED & confirmed** (`layering.browser.test.ts`; oracle = `elementFromPoint`):
1. z-index is scoped to a **stacking context**, not global — only elements in the *same* context
   compare; within one context, higher z wins. [Z3]
2. A context **contains** its descendants' z [Z1]: `isolation:isolate` (and `transform`, `opacity<1`,
   `filter`, `will-change`, `fixed`/`sticky`) trap child z inside; the context then competes as one unit.
3. **Cross-context, the contexts' own order dominates** [Z2]: `z:1` in a higher context beats `z:9999`
   in a lower one. "Who's on top" resolves hierarchically, context-by-context — *not a flat number
   line.* **Consequence:** the named scale only means anything *within one context*, so it **requires
   context discipline** — `isolation:isolate` at component roots — or the scale is a lie (a low-context
   `tooltip` loses to a high-context `base`).
4. **Footgun** [Z4]: `transform`/`opacity`/`filter` create contexts *accidentally* and trap z. Use
   `isolate` deliberately; watch for these creating one unintentionally.
5. **Host-beating constant** [Z5]: a max-int z beats a high host z **only at the root** [Z5a]; a
   *trapped* overlay loses despite max-int [Z5b] → content-script overlays must live at a **body-level
   root**, never nested inside a host stacking context. (Confirms Monky's append-to-body overlay setup.)

**Resolution — three tiers, because "what is on top" is not a declarable scalar.** The §4.4
reconstruction proved "on top" resolves *hierarchically*, context-by-context [Z2] — so there is **no
global maximum to query** and the "pick a bigger number" contest is unwinnable by construction (it
is the layering analogue of negotiated sizing: an irreducible global resolution no per-element
declaration can settle alone). The grammar therefore **splits layering into three tiers** and legislates
the boundary between them, rather than pretending one scale covers all:

- **Tier 1 — "above everything" (overlays, modals, popovers, toasts) → the TOP LAYER, not the z-scale.**
  The platform already maintains the "state to check against" you'd otherwise be guessing: the
  **top layer** (native `<dialog>.showModal()`, the Popover API) renders above the entire
  stacking-context tree **with no z-index at all**, ordered by a real engine-kept rule
  (last-opened-is-topmost). You don't *read* the registry, you **join it by opening** — which dissolves
  the contest by *refusing to play it*. So `overlay`/`modal`/`popover`/`toast` are **top-layer
  mechanisms**, not high z-numbers; prefer promotion over a scale value wherever the content can be
  promoted.
  > **Registry shape (this revision):** the tier-1 names are their own member set — **`top-layer-mechanism`**:
  > `overlay · modal · popover · toast` — with **signature: set-with-exclusivity** (not an ordered
  > chain — there is no "z order" among them; the engine's last-opened-is-topmost rule orders them at
  > runtime), **controls: none** (emits no `z-index`; the mechanism is `showModal()` / `popover` attr /
  > top-layer promotion), and **vocabulary: closed**. This is deliberately a *different axis* from the
  > tier-2 `z-scale`, so the linter never treats `modal` as a z rung. An author choosing `modal` is
  > choosing *promotion*, not a number.
- **Tier 2 — the structured in-page middle (`raised` · `dropdown` · `sticky` …) → the named ordered
  scale.** This is the only tier the declarative grammar truly owns, and it is *irreducible author
  intent*: "a dropdown sits below a modal" is a design decision, not a computable fact, so no platform
  primitive replaces it. The named scale governs **here and only here**, always within an
  `isolation: isolate` context (or it's a lie — [Z2]). This is the de-facto industry answer (semantic
  z-token ladders), independently reached by production design systems.
- **Tier 3 — the host-beating constant → the one justified big number.** Only for the
  content-script-in-an-unknown-host case (Monky's): when you cannot cleanly use the host page's top
  layer, a single boundary constant at a body-level root [Z5]. Exactly one number, quarantined, never
  part of the scale.

**Declarative/imperative boundary (legislated, for the spec).** Genuinely *querying* "what is currently
on top, place me above it" is **out of grammar scope** — it requires JavaScript, because the quantity
does not exist as a CSS-readable scalar. This is **not a gap in the grammar**; it is the seam where the
declarative layer stops and a runtime takes over (the layering analogue of the `place ∘ negotiate`
phase boundary). An automated consumer must **not** invent a CSS mechanism for dynamic top-querying —
there is none, by design; it routes to top-layer promotion (tier 1) or to JS, never to a coined property.

Still open `[RULING]` (Tier-2 values only, non-blocking, "no test decides"): the exact **tier-2
member list** (now *smaller* — tier-1 names move to top-layer mechanisms) and their z **ranges**;
the final spellings for `isolate` / `anchor` / `layer-<level>` / the tier-3 constant's value. These are
value/name decisions; the **three-tier structure and the declarative/JS boundary above are frozen.**

## 5. Skin — three kinds of structure (settled framing; rulings open)

Skin is **not** one thing and **not** all consensus.

### 5.0 — The grammar/theme seam (framing; non-structural)

**Theme is not a plane; it is the value-side of a seam that already runs through the grammar.**
`gap-comfortable` does not emit `16px` — it emits `var(--spacing-comfortable)`. Three things stack
there, three owners: the **step-name** `comfortable` (grammar — closed, stable), the **reference**
`var(--spacing-comfortable)` (the emitter's binding), the **value** `--spacing-comfortable: 16px`
(theme — swappable, defined once). The grammar owns *that there are six density steps and their names*;
the theme owns *what they are worth*; they meet at the `var()`. **That seam — not the layout/skin
split — is where grammar ends and theme begins**, and it already cuts through layout (spacing is
themed). So "skin" is not "the themed part." The real distinction is *how theme-bound a word is*, on
three rungs:

1. **Fixed** — grammar word, fixed CSS value (`horizontal`→`flex-direction:row`; `align-center`). Theme
   absent; the word *is* the value.
2. **Scale-bound** — grammar word (closed, stable step-name), value via `var(--scale-step)`.
   Spacing/size today; **radius** (→`border-radius`) and **type** (→`font-size`/`line-height`) are this
   rung (§5.1). Grammar owns the steps; theme owns the numbers.
3. **Fully open** — theme owns even the *word set* (`tone-*`, surfaces; §5.3). Grammar owns only the
   disjoint property and the `var()` socket; it enumerates nothing.

**Scope (consequence, not new law).** The skin the grammar specifies *now* is the rung-2 scales —
radius and type — emitted exactly as spacing is. Color/tone/ink are rung 3: declare the socket and the
disjoint property, defer the vocabulary and the values to the theme (§5.3). **Skin is the interface; a
theme is an implementation of it** — which is why a theme swap re-skins without touching a class.

### 5.1 Generative-proportional — *declare it; it's already math*
Ratios, modular scales, grids, rhythm, symmetry. Lawful, algebraic, author-set.
Earns value through **relational coherence** (everything derives from one system), not
from a preferred constant. This overlaps the spatial grammar (rhythm/scale — and the anisotropy
formerly named *proportion*, now per-side density).
- **Structure — FROZEN: scales are GENERATOR-DEFINED, not hand-listed.** A §5.1 scale is not a list
  of values; it is the **output of a declared generator** — a `(function, base, ratio)` triple that
  emits the steps (`value = base · f(n)`; for the geometric default, `base · ratio^n`). The values are
  coordinates; **the generator is the real object** (a process-relational stance: store the process,
  not the points). The current scales are ad-hoc and to be replaced by generated ones — spacing
  `4 8 12 16 20` is linear (arithmetic), radius `3 6 8 12 16 24` is irregular, type `12 14 16 18 21 24`
  is mixed; none *derive* from a system. This layer owns four such scales by **one method**:
  **spacing**, **radius**, **type** (`font-size`/`line-height`, moved here from the retired §4.1
  text/type axis), and **motion magnitudes** (`duration`/`delay`/`stagger`, §4.3, the same method in
  milliseconds). (Typeface/weight → §5.3; easing curves are a *closed vocabulary*, not a scale → stay
  in §4.3.)

- **Why geometric, not linear (the recommended default):** perception of size is ~logarithmic
  (Weber–Fechner) — the eye registers *ratios*, not differences — so a constant-*ratio* (geometric)
  scale reads as evenly-spaced while a constant-*difference* (linear) one reads as uneven (cramped at
  the top, sparse at the bottom). A modular scale is the visual analogue of a musical scale; the ratio
  plays the octave's role. So the default generator is **geometric, one global ratio**, which maximizes
  the relational coherence §5.1 is built on. But the *generator itself is a free parameter* — golden /
  Fibonacci, a denser two-stroke (interleaved progressions), or a custom perceptual curve are all
  admissible generators; geometric is the recommended default, not the only shape.

- **The specific generator is OPEN — and open *by intent*, pending EMPIRICAL measurement, not taste.**
  The `(function, base, ratio)` values are deliberately **not** picked here. They are the planned output
  of a **separate design-analysis project**: an algorithm that *reads/derives the generating function
  from existing designs*, to test in practice what shape these functions really have and reproduce them
  systematically. Picking a ratio now would be guessing at the very quantity that project will measure —
  the same oracle-not-derivation posture as the §6.4 flex reconstruction (characterize from observation;
  the measured function is authoritative, the assumed one is not). The slot is therefore frozen to
  receive a **measured characterization**, shaped to admit findings richer than a single ratio:
  - it may be **ratio-varying**, not constant (fit `log(value)` vs step-index; a pure modular scale is a
    *straight line*, slope = log r — the **residuals** from that line are the signal: where and how
    designers deviate from the math, e.g. geometric in the middle, compressed at the extremes);
  - it may be **per-property-family** (type often near-geometric; spacing lumpier under whole-pixel
    rounding; radius barely systematic — the current ad-hoc scales already hint at this);
  - deviations are **documented exceptions**, not silent — every hand-tuned escape from the generator is
    recorded, keeping the coherence honest.
  A recovered generator *is* a **layout equivalence class** (two systems with the same function but
  different parameters are the same proportional process) — so this project is the measurement instrument
  for the long-term layout-taxonomy vision, and §5.1 is where its output lands.

- **Late-bound / swappable (FROZEN):** because a scale is its generator, a theme can ship a *different*
  generator and the whole UI re-proportions coherently (everything reads the generated tokens) — a
  generator is a position in proportion-space the way a color theme is a position in taste-space. The
  grammar owns *that there is a generator*; *which* generator is swappable skin.

### 5.2 Perceptual response — *measure it*
Contrast/legibility/grouping. Lawful, measurable (e.g. contrast ratios). Mostly belongs
to skin tokens + a few rules.

### 5.3 Connotative / trend — *elicit it*
Color mood, ornament, "vibe." The consensus/learned layer; non-stationary and cyclical.
Keep **late-bound and swappable** (a theme = a position in taste-space). Out of scope for
this grammar; handled by the separate aesthetic-language effort (semantic differential /
repertory grid).

## 6. The compile/scale layer (LATER — do not start until §2–§5 are clean)

Once the vocabulary is pure and stable, the grammar **computes from one source** instead of
duplicating values. The foundation is **per-property classes over a shared scale** — general,
independent, single-sourced. (An earlier design used a runtime *value-channel*; it is **retired** —
see §6.3.)

### 6.1 The foundation — per-property classes over a shared scale (RULED L304)
One scale, defined once; one class per `property × density`, each reading the scale:

```css
/* ONE source: the density scale */
:root { --d-snug: 8px; --d-comfortable: 20px; --d-relaxed: 36px; }

/* general per-property classes — the full property name + the density word */
.padding-comfortable { padding: var(--d-comfortable); }
.margin-snug         { margin:  var(--d-snug); }
.gap-relaxed         { gap:     var(--d-relaxed); }
```
- **General** — every property, every density, any combination; `padding-comfortable margin-snug` on
  one element just works (independence is free — the common case the channel couldn't serve).
- **Single-sourced** — retune `--d-comfortable` once; every `*-comfortable` follows.
- **D×P classes, but generated** from `scale × properties` — the cost is CSS bytes, not authoring
  effort. (This is the proven atomic-scale shape; the grammar's value is the *semantic naming* + the
  plane/role structure *above* it, not a clever mechanism below.)

Naming — **full property name + density word** (`padding-comfortable`, not `pad-comfortable`): reads
like a sentence, no abbreviation ambiguity.

### 6.2 The key decoupling — single source ≠ single shared value
The retired channel conflated two separable things:
- **single source of truth** — comes from referencing one shared *scale variable* (`--d-comfortable`);
  plain per-property classes get this for free;
- **a single shared per-element value** — the channel's *constraint* (one rhythm per element), never a
  benefit.

Per-property classes keep the first and drop the second. That is the whole reason to prefer them.

### 6.3 RETIRED — the runtime value-channel
The earlier core technique set a custom property (`--space`) that property-agnostic *applicators*
(`gap`, `pad`, `margin`) consumed, for **D+P** classes. **Retired**, because:
- its headline win (single source) **did not need it** — a shared scale var delivers that generally;
- it only worked under a **constraint** — one rhythm per element, so `padding-comfortable margin-snug`
  (common) collided (see `docs/playground-value-channel.html`);
- its only *unique* residue was **modulation** (one `wide`/`tall` reshaping every applicator via
  `calc` on the live value; see `docs/playground-value-channel-wins.html`). A capability that only
  works under a constraint is not worth being the foundation — so the optional channel layer is **not
  kept** either.

`src/styles/grammar/valueChannel.browser.test.ts` now tests a retired mechanism → repurpose (to the
per-property foundation) or remove. (The CSS fact it relied on is still true and worth remembering:
custom properties **inherit by default**; a value you want element-local needs `@property {
inherits: false }`. Still applies to any *dedicated single-consumer* variable, e.g. `--card-radius`.)

### 6.4 Resolution order (size) — a *reconstructed*, falsifiable narrative
The orthogonal size inputs — m2 `grow`/`shrink`, m3 `basis`, and the `min`/`max` bounds — are
independent to *author* but **fuse into one computed width** at layout time. This is how they fuse.
It is **reconstructed from experiment** (the engine is the oracle), NOT read off the spec — a
spec-style hand-derivation got it wrong (see the correction below). Evidence/anchors:
`src/styles/grammar/flexPipeline.browser.test.ts` (E1–E4) + two clamp goldens.

1. **base** = `flex-basis` (auto → content size).
2. **free space** = container − Σ(**raw** base sizes) — *not* min/max-clamped.  *(E1)*
3. **distribute** free space: grow ∝ `flex-grow` (unweighted) · shrink ∝ `flex-shrink × basis`
   (weighted).
4. **freeze loop:** any item violating its `min`/`max` is clamped to that bound and *frozen*; the
   absorbed/freed space is recomputed and redistributed among the still-unfrozen; repeat until none
   violate.  *(E2 min-floor steals · E3 max redistributes · E4 cascades)*

**Clamp anchors (engine-verified goldens):** grow+max → `[150,150,200]`; shrink+min → `[100,70,70]`.

**Correction on record:** `min`/`max` do **not** pre-clamp the distribution baseline. The wrong
"step 2 → clamp basis to a hypothetical [min,max]" predicted `[175,150,175]`; the engine returns
`[150,150,200]`. Clamping happens **only in the freeze loop (step 4), after distributing from the raw
base.**

**Status: falsifiable, not derived.** If a browser/engine/spec ever diverges, one of E1–E4 turns red
and this narrative gets reconstructed again. Per the working rule: when verifiable evidence
contradicts the model, reconstruct the model to fit it — the engine is authoritative, the model is
the narrative that survives cross-examination.

## 7. Implementation scope

**RESOLVED (R9): membership must be established by *some* mechanism; the vocabulary stays unprefixed,
the *emitted* names get a deployment adapter.** §1's resolution law makes **membership** (stage-1)
the precondition for the unprefixed surface to be safe — the grammar's words (`content`, `selected`,
`raised`, `tooltip`, `horizontal`…) are deliberately collision-prone *human* names, safe only inside a
membership boundary. The ruling is therefore **per deployment**, all realizing the *same* membership law:
- **Shadow DOM deployment (Monky's case):** Shadow `:host` *is* the membership boundary — the
  unprefixed surface is scoped by the shadow root at runtime, no prefix needed. Monky mounts overlays
  at a body-level root inside a shadow tree (confirmed by §4.4 [Z5]). This is the primary target.
- **Light-DOM deployment (SSR, Astro, Markdown-rendered content, third-party app code):** there is no
  shadow boundary, so the unprefixed words **will collide** and membership must be established another
  way — a build-time **prefix** (`mk-`), a **cascade layer** (`@layer monky`), an **attribute scope**
  (`[data-monky] .horizontal`), or class **mangling**. This is a **deployment adapter**, not a change to
  the grammar: the *authored/human vocabulary is identical*; only the *emitted* class names differ.

So the rule is: **the vocabulary never changes; the membership mechanism is chosen per deployment.**
Shadow DOM is one realization (the one Monky uses); light DOM requires an explicit adapter rather than
relying on luck. An unprefixed surface with *no* membership mechanism is **unsupported** — that is the
one configuration the law forbids.

## 10. Testing — rules + laws, outcomes as canonical form

The flex config space is chess-sized (a single row of four ≈ **2.8 billion** outcomes), so tests
never enumerate it. Like chess, test the **rules** and the **invariants**, not the game tree. The
**numerical outcome** (the computed width-vector) is the *canonical form*: two configs with the
same widths are the same outcome, so reducing tests to numbers also quotients the redundancy.

**Two layers** (`src/styles/grammar/flexCharacter.ts`):

1. **Rule tests — static, jsdom, fast** (`flexCharacter.test.ts`, runs in the normal suite).
   Each token → the right CSS declaration: `compressible` → `flex:0 1 auto`, `basis-ratio` →
   `flex-basis:0`, `grow-2` → `flex-grow:2`. This is the first slice of the §6 compile mapping
   (`flexStyle()`). Catches "is the class defined right," and proves corners reduce to the
   `grow-N`/`shrink-N` primitives.

2. **Outcome tests — real layout, vitest browser mode** (`flexCharacter.browser.test.ts`).
   jsdom can't compute flex, so these measure real widths (`measureRow()`):
   - **Golden tables:** `(width, [tokens]) → [widths]` for canonical cases (the playground
     scenarios are ready fixtures) — e.g. `elastic ratio ×3 @600 → [200,200,200]`.
   - **Law / property tests:** invariants over many configs — conservation (Σw = W), grow ∝ grow,
     deficit ∝ shrink×basis (clamped at min), **ratio-invariance** (scale all grows by *c* → same
     widths), monotonicity, order-equivariance. These laws *are* the algebraic signatures, so this
     suite is **the constitution made executable**.

Setup for layer 2 (one-time): `npm i -D @vitest/browser playwright && npx playwright install chromium`,
then `npm run test:browser`. The default `npm test` excludes `*.browser.test.ts`, so layer 1 runs today.

## 10.1 — Predicate 4: static dimensional-purity check — ADDED & RUN this revision

> **Historical count, not current.** The "23 axes" figures below describe the registry as it stood
> *when this check was first added*. The registry has since grown to 33 axes (`registry.ts` is
> authoritative for the current count — see the top status block). The predicate itself, and the
> disjointness result it proved, still hold; only the axis count is dated.

§10's outcome tests prove the *negotiated* algebra. **Predicate 4 proves the *free* algebra's
precondition**: Law 3's claim that free-regime axes touch **disjoint** CSS property sets — the thing
that makes them commute. Until this revision the claim was confirmed only by inspection. It is now
**executable**: all 23 axes are populated with explicit `controls` (concrete CSS properties), and four
sub-predicates run statically over the whole set:

- **4a — self-consistency:** no axis controls a property it also forbids.
- **4b — cross-axis disjointness (free):** no two *free* axes control the same property, except the
  documented container/member **twin** on a facet-split property (only `display`: structure sets inner,
  m1 sets outer — CSS resolves them independently).
- **4c — free-vs-negotiated:** a negotiated axis may share with other negotiated axes (the flex solve)
  but must be disjoint from every *free* axis.
- **4d — state controls nothing:** state's defining invariant (condition-only).

**Result: the check found TWO real collisions that six component audits had missed — both now fixed;
re-run PASSES.** This is the value of a whole-set static check: it reads every axis at once and finds
property overlaps no single-component walk surfaces.

- **Collision 1 — motion stagger overwrote micro-delay** (`transition-delay` shared by motion-micro
  and motion-macro). *Not* a clean twin: same facet, so they overwrite. **Fixed** in §4.3 — stagger
  emits additive `--stagger`.
- **Collision 2 — `stretch` lived in two axes** (`align-self` shared by m3 and m4). Latent §4.1
  plane-mix. **Fixed** in §4.1 — m3 is `basis`-only; cross-axis stretch is m4.

After both fixes: 23 axes (21 free, 2 negotiated), the only shared free property is the `display`
twin. **Dimensional purity holds — true-after-correction.** First executable confirmation of Law 3,
and it arrived with two bug-fixes the constitution needed.

**Artifacts & how to re-run.** The check is the session script **`full-registry.ts`** (populates all
23 axes, runs sub-predicates 4a–4d, prints the property-ownership index); the collision diagnosis and
fix-proofs are **`collision-analysis.ts`**; the four well-formedness predicates the §9.1 audits use are
**`registry-lint.ts`**. All three run under `npx tsx <file>` and pass as described. **Status: session
scripts — NOT yet committed** to `src/styles/grammar/`.
